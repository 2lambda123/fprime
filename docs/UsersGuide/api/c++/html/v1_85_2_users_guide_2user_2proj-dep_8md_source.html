1 # Projects and Deployments
2 
3 F´ is naturally organized into projects and deployments. Projects use F´ to solve some embedded system problem, and may
4 consist of one or more deployments. Deployments are tied to individual builds of code and are typically used to drive
5 a single device.
6 
7 A topology is a specific set of interconnected components, which represents the design of a deployment. Further
8 explanation of projects and deployments and how they differ is explained in this guide.

21 ## Projects
22 
23 A project may consist of one or more related deployments. Each deployment is one instance of F´ software. Projects are
24 used to organize F´ code that is closely related. They define at least one deployment (see above), but may define
25 multiple deployments. Component designs and source may be shared between these related deployments.
26 
27 There are several reasons a project may contain multiple deployments.
28 
29 1. A project consists of multiple spacecraft/electronic platforms/CPUs/systems. F´ can be used on multiple nodes such
30 systems.
31 2. A project has test deployments, mocked-deployments, or other deployments that allow for specific testing setups.
32 
33 Figure 1 shows an example of a project, the Mars Helicopter flight software, which was implemented with F´. The Mars
34 Helicopter is the project, but has two deployments: one designated for the helicopter, and one designated for the base
35 station allowing the.

37 ## Deployments
38 
39 Deployments are tied to builds; for each build of the code there is a single deployment. These deployments may share
40 components and ports from within the project, but the topology and the individual build of F´ are typically unique.
41 
42 Deployments may define custom components and ports that are only used in that specific deployment. They may also inherit
43 other components, like those defined by the F´ framework. Each F´ deployment uses a set of components and ports,
44 and has a single topology. Components and ports are organized into the deployments topology to represent the behavior of
45 that deployment.
46 
47 Deployments contain the needed build system artifacts to build the F´ framework, components, ports, and the topology
48 into an executable that can be deployed onto embedded hardware, or even run on the user’s computer.
49 
50 
51 In our example multiple non-homogeneous systems are defined for the Mars Helicopter. It has multiple deployments as
52 each system would have a separate unique executable to control it.

55 ## Topologies
56 
57 A topology is a specific set of interconnected components that represent a system. Topologies contain instantiations of
58 each component, and list connections between the ports of the components.
59 
60 Projects contain one or more deployments that are closely related, as discussed above, while deployments contain a
61 that represents the design of the system including, ports, and components needed for the specific use of F´.
62 
63 ## Conclusion
64 
65 Projects are containers of related deployments to solve some problem. Deployments are a container for a topology and
66 build configuration to create an executable for a specific device used on the project.
