21 ## F´ Terminology
22 
23 This section will cover basic terminology used in this and other tutorial with respect to F´. It may be used as a
24 reference for keywords that have specific meaning in an F´ project.
25 
26 **Port:** a port represents a connection between *Components*. These act as communication channels in F´. This allows
27 *Components* to interact with each other. A set of *Ports* acts as a defined interface to a *Component*.
28 
29 **Component:** the location of program behavior. Components execute the software logic of the system and typically
30 define an interface of *Ports* that act allow them to interact with other *Components*. Components additionally
31 may define *Commands*, *Events*, *Channels*, and *Parameters*, which define standard data patterns in F´.
32 
33 **Topology**: a topology represents the full F´ system. It is a set of *Components* and the connections between those
34 *Components*. The *Topology* specifies the full system and all the communication channels.
35 
36 **Event:** an event, also known as an EVR, is a type of downlinked data that represents a single event in the system.
37 An event is composed of a name, format string, and set of arguments that described what happened. Events represent a
38 history of the system. Events are defined per-*component*.
39 
40 **Channel:** a channel, also known as a telemetry item, is a single value read and downlinked. Channels are consist of
41 a type, a format specifier, and a value. Channels represent current system state. Channels are defined per-*component*.
42 
43 **Command:** a command is uplinked data items that instructs the system to perform an action. *Commands* consist of an
44 opcode, a mnemonic, and a list of arguments to the command. Commands are defined per-*component*.
45 
46 **Deployment:** a deployment is a single instance of F´. Although a single F´ project can use/define multiple
47 deployments, a deployment represents one build/executable.
48 
49 **Build Directory/Build Cache:** this directory, created by the `fprime-util` described below, is required to build
50 CMake projects. It stores all by-products, generated code, and build products from an F´ build.
51 
52 **Ground System:** the computer system running to collect the data from the running F´ deployment. There is a basic
53 ground system supplied with F´ for development purposes.
54 
55 **Toolchain:** a set of build tools to build for a specific architecture. i.e. the "raspberrypi" toolchain will build F´
56 deployments for the arm/Linux architecture of the raspberry pi.

58 ## F´ Development Process
59 
60 F´ development typically starts with creating new Ports and Components as these can be built and tested as units and
61 then the full F´ system deployment topology can be assembled for integrated systems testing and release/operations. This
62 can be done iteratively by creating a few Components then integrating them into the system topology and then repeating
63 with the next most critical components.
64 
65 F´ development is a design-first approach. To start, a developer should design any unique ports that the Component will
66 need. If desired, the port can be built and compiled to ensure that the design is correct. Since the port will be
67 entirely auto-generated, developers usually refrain from unit testing the port and defer unit testing to the Component.
68 
69 Next, developers design components. Components are designed by specifying the ports that represent the Component's
70 interface and the Commands, Events and Channels that the Component handles. This tutorial focuses on the tool suite, so
71 the design is already in-place. Users can see the design by inspecting Components in the `Ref` directory. Creation of
72 new components from design through test is demonstrated in the [Math Component Tutorial](../MathComponent/Tutorial.md).
73 
74 The next step for developing a Component is to implement the code. To do this, one can generate template stubs to fill
75 with the developer's stubs. To do this, a user runs the F´ tool suite to generate these stubs. Then implements, builds,
76 and tests from there. Unit tests template stubs are generated in the same way, and should test-driven development be
77 preferred, these tests stubs could be generated and implemented before the Component's code.
78 
79 Finally, the Component is integrated into the deployment and the entire deployment may be built and installed.
80 
81 This tutorial will walk developers through generating the templates, building example Components, and
82 building/installing the full deployment. It will use the `Ref` application as an example for this.

84 ## Working the F´ Development Process with `fprime-util`
85 
86 In order to ease in development, the F´ team has created a small wrapper for the CMake build system. This will recover the
87 advantages of previous F´ development patterns, while allowing for the power and configurability of CMake as an underlying
88 build tool.
89 
90 ### Build Cache Generation
91 
92 In order to run CMake, a build cache needs to be generated. This is typically done once for each platform the user wishes
93 to compile against. It is usually done right after F´ is checked-out and repeated for each platform as they are added. It
94 is a setup step and isn't formally part of the F´ development process.
95 
96 To run this tool, the developer will use the `generate` subcommand. It takes one optional argument: the toolchain file
97 used in CMake to compile for a specific platform. If not supplied, the `native` toolchain will be used and F´ will be
98 setup to run on the current platform (typically Mac OS, or Linux depending on the developer's chosen OS).
99 
100 **Generate the Ref Application for Native Compilation**
101 
102 To generate a native build, follow the commands shown below. It is assumed that the user has checked out the fprime repository
103 and is performing commands relative to the location of that checkout.
104 
105 ```
106 cd fprime/Ref
107 fprime-util generate
108 ```
109 
110 Here the utility will create two standard builds: one for the standard build, and one for the unit test build. This will enable
111 the user to take advantage of all parts of F´ without generating their own CMake build caches. **Note:** should a developer
112 be developing or improving F´ provided infrastructure components, then the `generate` command should be run in the F´
113 root directory. However, most developers do not need this functionality.
114 
115 **Generate Cross-Compile of the Ref Application for Raspberry PI Platform**
116 
117 Most developers wish to run F´ on embedded hardware. This is done by generating a cross-compile using a different CMake
118 toolchain by providing the toolchain argument. The above invocations assume the default "native" toolchain.
119 
120 Here, the "raspberrypi" toolchain supplied by F´ is used. This will generate a cross-compile for the raspberry pi
121 architecture. The user should ensure to follow the Raspberry PI setup steps found at: [RPI](https://github.com/nasa/fprime/blob/master/RPI/README.md)
122 ```
123 cd fprime/Ref
124 fprime-util generate raspberrypi
125 ```

127 ## Creating Implementation Stubs
128 
129 Once the build system cache is generated, developers will need to generate the stubbed implementations of a Component's C++
130 files. This can be done using the `impl` command of the F´ utility. These commands assume a default build has been generated.
131 Typically passing in a toolchain is unnecessary as these templates are platform agnostic.
132 
133 **Generating Implementation Stubs of SignalGen**
134 ```
135 cd fprime/Ref/SignalGen
136 fprime-util impl
137 ```
138 
139 This creates two files for the component. These are `<Component>Impl.cpp-template` and `<Component>Impl.hpp-template`.
140 The user can then rename these files to remove `-template`. The file is then ready for C++ development. By generating
141 `-template` files, we won't accidentally overwrite any existing implementation should the developer need to repeat this
142 step.
143 
144 ## Creating Unit Test Implementations
145 
146 Once the build system cache is generated, developers will need to generate the stubbed implementations for creating
147 unit tests for a Component's C++. This can be done using the `impl --ut` command of the F´ utility. These commands
148 assume a default build has been generated. They can be run passing in a toolchain, but this is typically not done because
149 they are toolchain agnostic.
150 
151 **Generating Unit Test Stubs of SignalGen**
152 ```
153 cd fprime/Ref/SignalGen
154 fprime-util impl --ut
155 ```
156 
157 This creates the following files, that are typically moved to a sub folder called `test/ut`. The files created are
158 placed in the current directory and named:
159 ```
160 Tester.cpp
161 Tester.hpp
162 GTestBase.cpp
163 GTestBase.hpp
164 TesterBase.cpp
165 TesterBase.hpp
166 TestMain.cpp
167 ```

169 ### Building Components and Deployments
170 
171 Once a developer has implemented a component, it is time to build that component and test that component. In order to build
172 a component the `build` subcommand needs to be run. This will build the "current directory" that the developer is in.
173 That means that should the user change into a deployment directory then the build command will build the full deployment.
174 Should the user desire to build a component, navigate into the Component's directory and run the build command.
175 
176 **Build SignalGen Component**
177 ```
178 cd fprime/Ref/SignalGen
179 fprime-util build
180 ```
181 
182 The user can also build a deployment by navigating to the deployment directory and running the `build` command. This is
183 shown below:
184 
185 **Build Ref Deployment**
186 ```
187 cd fprime/Ref
188 fprime-util build
189 ```
190 **Note:** the user almost always wants to run the "install" command on deployments. This builds the binary like "build"
191 but also copies the binary to the deployments "bin" directory, i.e. `fprime/Ref/bin`. Install is described below.
192 
193 This process also built the Dictionaries for the project and places the dictionary in the "Top" folder of the deployment.
194 This happens any time the "install" or "build" command are run on a deployment.
195 
196 The user can also build a component or deployment for a cross-compile by specifying the toolchain. A previous generate
197 for that toolchain should have been run. Again for deployments, the user typically should run "install", see below. These
198 steps require the setup described here: [RPI](https://github.com/nasa/fprime/blob/master/RPI/README.md)
199 
200 ```
201 cd fprime/Ref/SignalGen
202 fprime-util build raspberrypi
203 cd fprime/Ref
204 fprime-util build raspberrypi
205 ```

207 ## Installing the F´ Executable and Dictionaries
208 
209 Once the deployment is built, it would be nice to be able to install the binary and dictionaries. This will enable the users to
210 quickly find and run the deployment. This installation can be run using the following command. Everything will be installed to
211 the directory defining the deployment. i.e. `fprime/Ref/bin`. Install will also invoke `build` so users should use this in-place
212 of build for deployments.
213 
214 **Installing the Ref Deployment and Running the Binary Assuming Linux**
215 ```
216 cd fprime/Ref
217 fprime-util build
218 ./bin/Linux/Ref # Run the deployment
219 CTRL-C # Exit the application
220 ```
221 Running the application as part of the development ground data system is shown below.
222 
223 The user can also install a cross-compile.
224 ```
225 cd fprime/Ref
226 fprime-util build raspberrypi
227 ```

229 ## Building and Running Unit Tests
230 
231 Unit tests can be build using the the `build --ut` command of the `fprime-util`. This will allow us to build the unit tests
232 in preparation to run them. The user can also just run `check` to build and run the unit tests.
233 
234 Before building unit tests, the unit test build cache must be generated:
235 
236 ```
237 cd fprime/Ref
238 fprime-util generate --ut
239 ```
240 
241 **Building Unit Test of SignalGen**
242 ```
243 cd fprime/Ref/SignalGen
244 fprime-util build --ut
245 ```
246 
247 Once built, the unit test can be run using the following command. **Note: this will produce a message warning of no
248 tests run because SigGen has no unit tests defined.**
249 ```
250 cd fprime/Ref/SignalGen
251 fprime-util check
252 ```
253 The user can also build the unit test, but must copy it to the hardware and run it here. This can be done with a cross-
254 compile by running the following commands.
255 
256 **Cross-Compile Unit Test of SignalGen**
257 ```
258 cd fprime/Ref/SignalGen
259 fprime-util build raspberrypi --ut
260 ```
