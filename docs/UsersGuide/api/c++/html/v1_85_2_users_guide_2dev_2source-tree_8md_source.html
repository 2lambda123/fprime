21 ## Autocoders
22 
23 The Autocoders directory contains the scripts that are used to generate
24 source for the components. It is implemented as a set of Python 2.x
25 scripts that process the XML files used to describe the various entities
26 in the system. The directory also contains the C language version of the
27 hierarchical state machine (HSM) framework that has been flown on
28 several missions at JPL. The HSM is not needed by the framework or the
29 code generation, but is available for use by developers writing
30 component adaptations if they wish to use HSM for implementing state
31 machines. Developers do not need to study the implementation of the
32 autocoder in order to utilize it. The following directories are of
33 interest to developers:
34

67 ## Fw
68 
69 The Fw directory is the location of framework code and base classes.
70 This code should not be modified (with one exception, see Cfg in Section
71 5.3.1) by developers using the framework. The code generation relies on
72 the types declared to construct the entities in the architecture.

74 ### Cfg
75 
76 The Cfg directory contains a header file (Config.hpp) that is used to
77 configure various properties of the architecture. The developer can
78 modify the file to tune the architecture for the requirements of a
79 particular deployment environment. The contents are described in Section
80 9.
81 
82 The file AcConstants.ini contains a set of values for variables used in
83 the code generator. This file follows the Python ConfigParser syntax,
84 which is based on Windows .ini files. Using this file allows component
85 features like opcodes and port numbers to be changed without modifying
86 the component XML itself. See the component XML specification in Section
87 6.6.3.

281 ## Svc
282 
283 The intent of this directory is to provide a set of components
284 implementing services that would be useful for a flight application. The
285 service layer in software is traditionally the layer that provides the
286 mechanism for executing the software and managing data. The components,
287 ports, and other types are examples of how the architecture can be
288 applied. The component example implementations are very simple; flight
289 versions would most likely be more sophisticated. The way development is
290 done is that the developer will define the components and their
291 properties in XML. The code generator will generate C++ classes that
292 encapsulate the features of the component. The developer will then write
293 a class the derives from those generated classes and implement the port
294 methods. For these directories, each file will not be described, but a
295 higher level description of what each directory contains will be given
296 instead. The descriptions are as follows.
297 
298 ### ActiveLogger
299 
300 This directory contains a component XML description and implementation
301 for an active component that accepts serialized log events. The input
302 port accepting log entries puts them in a message queue for the
303 component thread. The component thread calls the port handler in the
304 derived class written by the developer. In this case, the handler simply
305 takes the log entry, serializes it into a communications buffer, and
306 sends it out via the output communications port. There are commands for
307 filtering the event levels.
308 
309 ### ActiveRateGroup
310 
311 This directory contains a component implementation of an active rate
312 group. In real-time programming, a rate group is a thread of execution
313 that does a sequential set of operations that execute cyclically and are
314 required to be complete by a certain deadline. In this case, this
315 component provides the thread for the rate group and sequentially calls
316 a set of output ports that would be connected to other components doing
317 the specific operations that are required.
318 
319 ### CmdDispatcher
320 
321 This directory contains an implementation of a command dispatcher. The
322 commands are received in serialized form from another component. The
323 command identifier (opcode) is deserialized, and an output port is
324 looked up that is matched with that identifier. The port is then invoked
325 with the serialized command arguments. After executing the command, the
326 component responds back to the dispatcher via a response port and
327 reports on the outcome of the command. The dispatcher then calls a
328 status output port if connected in the event there was something else
329 such as a sequencer waiting for a result. During initialization, a
330 command registration port is called by components to match the
331 identifier with the port that the component is executing commands from.
332 The component also implements some NO\_OP commands.
333 
334 ### CmdSequencer
335 
336 This directory contains an implementation of a command sequencer. A
337 sequence file uploaded to the system contains a set of commands that are
338 executed in order with optional time points. The sequencer waits until
339 the current command is complete before executing the next in the
340 sequence. A failed command terminates the sequence.
341 
342 ### CmdRecord
343 
344 This directory specifies the port used to pass command buffers between
345 the SequenceFileLoader and the SequenceRunner components.
346 
347 ### Cycle
348 
349 This directory specifies the port used to drive the ActiveRateGroup
350 components. The port passes a time stamp indicating when the cycle was
351 started.
352 
353 ### Fatal
354 
355 The directory specifies a port used to pass a notification that a FATAL

359 ### GndIf
360 
361 The directory contains just the XML definition of a component that could
362 be used to send and receive communications packets. The uplink port
363 would be connected to the command dispatcher for executing commands, and
364 the downlink port would be connected by components collecting downlink
365 telemetry like logs and channelized data. A derived class implementing a
366 TCP/IP socket version can be seen in SocketGndIf.

368 ### Hub
369 
370 A hub is a pattern for communicating between computing nodes. A hub
371 component is a component that contains input and output serialized
372 ports. As mentioned in Section 5.3.4, when a typed port is connected to
373 a serialized port, the port arguments are serialized and passed as a
374 data buffer to the serialized port. Likewise, when a data buffer is sent
375 from a serialized port to a typed port, the data are deserialized back
376 into the typed arguments of the port. A hub component takes the
377 serialized data passed to it and sends it via a communication channel to
378 a hub on a remote node. The remote hub takes that data and calls a
379 serialized output port connected to a typed port of the same type as the
380 original port. That allows components to be interconnected across
381 computing nodes without any modifications to the components themselves.
382 This particular hub is implemented as an active hub, which means that
383 the serial buffers from the incoming ports are put in a message queue
384 and then sent out the DataOut ports on the thread of the component. The
385 data output ports are in the form of a generic com buffer. The data
386 output ports would be connected to a component that manages the
387 communication hardware. Likewise, incoming data on the DataIn port is
388 queued for the component thread, which sends it out via a serialized
389 output port.

489 non-active array to the packet output port. If incoming telemetry calls
490 happen during the copy operation, they are placed in the active array.

492 ## Os
493 
494 This directory contains classes that abstract operating system features.
495 This allows the components that are code generated to not be dependent
496 on a particular operating system. The architecture is dependent on these
497 classes. The subdirectories contain implementations of the class for
498 different operating systems. Not all operating systems will implement
499 all classes. The classes are as follows.
500 
501 ### Task
502 
503 This class represents a task (AKA thread) in an operating system
504 process. It has methods for starting, ending, waiting, and suspending
505 tasks. This class is used by active components.
506 
507 ### Queue
508 
509 This class represents a message queue. It has methods for creating,
510 writing to, reading from, and destroying queues. This class is used by
511 queued components.
512 
513 ### Mutex
514 
515 This class represents a mutex. It is used to guard critical sections of
516 data and code. It is used by components that have guarded ports (see
517 architecture description).
518 
519 ### File
520 
521 This class represents a file. It is used to abstract away various
522 operating system implementations of file I/O.
