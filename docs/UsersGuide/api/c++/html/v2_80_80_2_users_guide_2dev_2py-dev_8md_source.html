37 ## When and How to Format Your Code
38 
39 Everyone has their own habits and style when writing code. We are asking you to change your habits, but this is made
40 easy through the use of a code formatter. This ensures your code is consistent with F´ so that reviewers and
41 collaborators can more quickly comprehend what the code is doing. This can easily be done executing the `black`
42 formatter right before you commit and push your code.
43 
44 Here is how to run black:
45 
46 ```bash
47 # format a single file
48 black example-file.py
49 # format an entire folder and subfolders
50 black path/to/folder/
51 ```
52 
53 ### Git Pre-commit hooks
54 
55 If you followed the Development Install Instructions above you will have automatically set up pre-commit hooks.
56 These will run automatically whenever you commit your code. Currently the only pre-commit hook we have is `black`.
57 Black automatically formats your python code so that it is consistent with the rest of the F´ python code.
58 
59 ```bash
60 $ git commit -m "Add new python code"
61 Format Python Code (black)...............................................Passed
62 [master 3415f7dd] Add new python code
63  1 file changed, 30 insertions(+)
64 ```
65 
66 If black finds unformatted code the commit will fail, meaning that you will have to stage the formatted changes and
67 run the commit again. If for some reason black's formatting introduces a bug into your code you can optionally skip the
68 pre-commit hook by modifying the 'SKIP' environment variable in bash. For example:
69 
70 ```bash
71 SKIP=black git commit -m "foo"
72 ```
73 
74 However this should only be used in an emergency. If SKIP is used for all of your commits it will dramatically
75 reduce code quality over time.
76 
77 ### Why did we choose black
78 
79 Black is an opinionated formatter that will format your code without any configuration or input from the programmer.
80 Why did we choose an “opinionated” formatter? In short, convention is so much more efficient than configuration. With
81 convention, everyone just does what is dictated (in this case from the tool) and there is no room for choice, which
82 causes entropy in the project (e.g. one developer chooses spaces, another chooses tabs, and chaos results). As entropy
83 increases, the code is harder to understand automated tooling stifled, and development becomes more difficult. Hence,
84 projects select coding styles, standards, best practices, etc. as a way to reduce entropy. By accepting an established
85 opinion, we can gain the advantages of a well-ordered project via a tool. Assuming the opinions of the tool are not too
86 onerous, we’ll have a better project as a result. Be like the Borg and just assimilate...

88 ## Static Analysis
89 
90 While almost all projects require unit tests, not enough them consistently use static analysis to improve code quality.
91 With a few simple commands static analysis will drastically reduce time wasted with simple errors or issues and increase
92 the lifecycle of your code.
93 
94 F uses Pylama to perform static analysis on our python code. Pylama contains pylint, pyflakes, pycodestyle(pep8),
95 mccabe, pydocstyle, radon, gjslint, eradicate, and Mypy. Combining the output from the tools that you want into a single
96 file.
97 
98 ```bash
99 # How to run pylama
100 pylama -o path/to/fprime/pylama/setup.cfg path/to/directory/you/want/to/analyze/
101 # the '-o' specifies the configuration file
102 ```
103 
104 Pylama can be configured within the setup.cfg file. This file can also go in the directory you are analyzing if you wan
105 a different configuration for the python files in that directory. Inside the configuration file you can configure both
106 pylama as a whole and each of tools individually.
107 
108 The recommended F´ pylama config is at: Fw/Python/pylama.cfg
