1 # Application-Manager-Driver Architecture
2 
3 Standard F´ applications typically break down into various layers. This is done to apply standard software layering
4 techniques to F´ applications. Each layer is dependent on layers below it, however; these layers do not depend on layers
5 above them. These layers are composed of individual modules (usually **Components**) that represent concrete units of
6 functionality. These units have defined interfaces for communicating with other modules. In F´ these interfaces are the
7 list of **Ports** the component uses.

18 ## A Brief on Software System Architecture
19 
20 The software system architecture is designed to break down software into modules. These modules provide separation of
21 function, definition of interfaces, behavioral characteristics, testing at the unit level, and ownership. These modules
22 are known as **Components** in F´ and all external functions of the component are defined as an interface for
23 interacting with the component. In F´ the component’s **Port** list is its external interface. In addition,
24 initialization functions and Constructors allow for setup and construction from the main thread.
25 
26 ### Software Layering
27 
28 Software layering allows separation of concerns by implementing logic such that a module only interacts with the layer
29 below it. In addition layering allows for decoupling code dependencies, as upper layers depend down the later stack, but
30 layers do not depend up the stack. Thus preventing circularly dependencies.
31 
32 Layers can be replaced to support reusability, simulation, and test. Fault protection can be isolated to a single layer
33 and thus handled at the appropriate level.
34 
35 ## Application Manager Driver Pattern
36 
37 The F′ component architecture comfortably fits into a layered software architecture. This can be done using an F´ design
38 pattern known as the Application Manager Driver pattern. F´ applications are assembled into layers of components. There
39 are typically three layers to this design pattern:
40 
41  1. application
42  2. manager
43  3. driver
44 
45 These layers are further discussed below using the example shown in Figure 1. Each layer defines components for only
46 that layer’s functionality.

53 ## Application Component
54 
55 The application layer implements mission specific functionality. It is the layer that defines cross-cutting
56 functionality that links together many parts of the system. However, it is not privy to the details of the parts of the
57 system. It represents system logic and uses peripherals and other components to perform the detailed behaviors of the F´
58 application.
59 
60 In our example, the application level performs the function of a robotic arm, however; it only knows that the arm is
61 composed of servos and the high-level interface to those servos, but knows nothing of the hardware interface to
62 run those servos nor how to translate high-level servo interface into data packets.
63 
64 ## Manager Component
65 
66 The manager layer manages a particular peripheral by using the interface driver at the abstract level (through the
67 driver’s interface of ports). The manager does not know how the peripheral will be used by the application. It does not
68 even know that it is used at all by the Application. However, the manager does know of the driver that is used
69 to talk to its peripheral and how to translate high-level commands into driver messages.
70 
71 In our example, a servo manager is defined to know how to control servos through a driver and provides a generic
72 high-level servo interface to any application that might use servos.
73 
74 ## Driver Component
75 
76 The driver layer is dedicated to a particular hardware interface. It is written to interact with only that type of
77 device, and does not need to know the use of hardware connected at the other end of the driver.
78 
79 In the example, the driver provides an interface to the hardware device (I2C), but not an interface to the servo. It is
80 up to the manager to know how to control the servo via the driver. In our example the driver runs an I2C bus but doesn't
81 know what I2C hardware it is talking to.
82 
83 ## Conclusion
84 
85 By layering F´ projects using the Application Manager Driver pattern, standard software architecture techniques can be
86 adapted to an F´ project. In complex systems there are often more components (multiple drivers, multiple managers) at
87 each layer, but these components should still fit into a single layer.
