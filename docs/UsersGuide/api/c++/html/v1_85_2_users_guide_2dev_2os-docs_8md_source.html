1 # Operating Abstraction Layer
2 
3 The framework provides OS services abstraction classes to allow developers to write code in a more portable fashion.
4 Implementations of these classes are provided for Unix variants (POSIX), and Mac OS. Additional ports can be done by
5 providing additional implementations as a new OS is added. This guide will walk through the critical files in this
6 layer to enable developer understanding and give an overview for individuals seeking to port to a new OS.

20 ## Tasks
21 
22 Tasks are called threads under Unix variants, and tasks under other OSes. A
23 developer may wish to start tasks to wait on an event or a resource that
24 is not a port invocation. The Os task class definition can be found in
25 **Os/OsTask.hpp**. The OS implementations are found in various
26 subdirectories. Active components require use of tasks to implement their
27 features. Table 23 describes the methods of the class.
28

31 | Method | Description | |
32 | ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
33 | start() | Starts the task. The arguments are: | |
34 | | **Argument** | **Description** |
35 | | name | String name of the task. |
36 | | identifier | A user-selected unique integer identifying the task. |
37 | | priority | The priority of the task: 0 is lowest, 255 is highest. |
38 | | stackSize | The size of the stack, in bytes. |

55 ## Task Registry
56 
57 Task registries are meant to be used to track all task instances in the
58 system. The concept is that this registry would contain pointers to all
59 the instances and be able to iterate over them and perform actions. The
60 file **Os/Task/Task.hpp** provides a base class definition of a registry
61 with the addTask() and removeTask() pure virtual function definitions.
62 Developers would write their own derived classes to implement a registry
63 that is appropriate for their system.

106 ## Interval Timer
107 
108 The class definition can be found in Os/IntervalTimer.hpp. An interval
109 timer is a facility that starts and stops the measurement of passage of
110 time in the system. It is not an expiration timer that signals when it
111 is complete. It is used to measure execution times of, or to timestamp,
112 software activities. Table 25 provides the methods and their
113 descriptions.
114 
115 **Table 25.** Interval timer method
116 descriptions.
117 
118 | Method | Description |
119 | --------------------- | ---------------------------------------------------------------------------------------------------- |
120 | start() | Saves the start time of the timer. |
121 | stop() | Saves the stop time of the timer. |
122 | getDiffUsec() | Returns the time difference between start and stop in microseconds. |
123 | getDiffUsec() | Overloaded version that takes two times, subtracts them, and returns the difference in microseconds. |
124 | getRawTime() | Gets the current raw time value. Can be used for time-tagging events. |

152 ## Interrupt Lock
153 
154 The class definition can be found in Os/InterruptLock.hpp*.* An
155 interrupt lock prevents interrupts from preempting code execution. It
156 can be used as a very lightweight mutex on platforms that support
157 interrupt locking, but should be used carefully as it is not subject to
158 priorities like a conventional mutex. In addition, it defers interrupts
159 that could be time critical, so the code executed between the locking
160 and unlocking should be very short in duration. This is a portable
161 interface, but the user should be aware of the behavior for each OS.
162 Table 27 provides the methods and their descriptions.
163 
164 **Table 27.** Interrupt lock method descriptions.
165 
166 | Method | Description |
167 | -------- | -------------------------------------------------------------------------------- |
168 | lock() | Lock interrupts. |
169 | unlock() | Unlock interrupts. |
170 | getKey() | Returns the interrupt lock key, if used by the OS. This is typically not needed. |

209 ## Log
210 
211 This class definition can be found in **Os/Log.hpp**. It is an interface to
212 a system logging facility. It is meant to abstract the VxWorks logging
213 facility. It is a subclass of `Fw::Logger` and thus must be registered after construction. Compiling in
214 **Os/LogDefault.cpp** into your deployment will automatically creat an **Os::Log** and register it.
215 Table 30 provides the methods and their descriptions.
216 
217 **Table 30.** Log method descriptions.
218 
219 | Method | Description | |
220 | ------ | -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
221 | logMsg | Log a text message. The arguments are: | |
222 | | **Argument** | **Description** |
223 | | fmt | Format string to fill and print. |
224 | | a1 to a6 | Values to be printed. They will be inserted into the format string based on the format string specifiers. The type of the argument is POINTER\_CAST (normally an integer), but a typical usage is to cast the values to display to POINTER\_CAST and allow the format string extraction to get the correct value. This can include strings (char\*), but the location in memory that holds the string must persist since the format string may be printed on another task in the system. |
