14 ## Baremetal Systems
15 
16 A baremetal system is a system that does not run an Operating System to support the F´ software. Thus, the F´ software
17 must provide for basic services such as filesystems, thread schedulers, etc, to run. Since F´ was originally run on
18 systems that provide these services, a number of precautions must be taken when designing F´systems for baremetal
19 platforms.
20 
21 ### The Joy of Passive Components
22 
23 First and foremost, baremetal F´ systems should avoid using **Active Components** at all costs because these components
24 require quasi-asynchronous execution contexts in which to run. i.e. they need thread such that they can execute in
25 "parallel" with each other. **Note:** if you **must** use **Active Components** you should thoroughly review the
26 [thread virtualization](#thread-virtualization) section of this document an associated technology.
27 
28 If your system can be entirely defined by **Passive Components** then implicitly every port **invocation** would be
29 synchronous and the the execution context would be entirely delegated to every component. A discussion of the source
30 of that delegated execution context comes next.
31 
32 ### Choosing an Execution Context
33 
34 Since the OS is not around to execute F´, the implementer of the F´ project must choose an execution context for F´ to
35 run on. That is, ensuring that some call invokes all of the **Components** that compose the F´ system. Otherwise some
36 components will not run. Typically, this is handled by composing an F´ baremetal system into components that are all
37 driven by [rate groups](../best/rate-group.md). Designing the system this way ensures that all execution is derived from
38 one source: the rate group driver and thus reducing the problem to supplying an execution context to the rate group
39 driver at a set rate. All calls needed will execute a during a sweep through the rate groups a their derived rates.
40 
41 **Note:** other options exist (see [Thread Virtualization](#thread-virtualization) below).
42 
43 Although a full discussion of supplying execution context to the rate group driver is outside the scope of this
44 documentation, this author wanted to provide a few tip. First, F´ execution should be primarily derived from the main
45 program loop. i.e. embedded software typically looks like the following and the loop-forever `execute();` action should
46 trigger the rate group driver at a set interval.
47 
48 ```C
49 // Run once setup
50 setup();
51 
52 // Do this forever
53 while (true) {
54  execute();
55 }
56 ```
57 
58 Now all that is required is to determine when this interval has elapsed. This can be done spinning on a hardware clock
59 signal, calculating elapsed time by reading of clock registers, using timing library functions, the `sleep()` call, or
60 by an timer driven interrupt service routine (ISR). **Note:** ISRs are complex items and should be studied in detail
61 before going this route. Notably, the ISR should not execute the rate group directly, but rather should set a flag or
62 queue a start message and allow the `while (true) {}` spin in the main loop to detect this signal and start the rate
63 groups.

65 ## Multi-Core and Multi-Device Systems
66 
67 We have yet to see any issues running F´ on multi-core systems. Some users have been successful scheduling high-priority
68 components to designated cores. In general these systems behave just fine. **Note:** some portions of F´ use `U32`
69 types so synchronize between threads. In many systems this is a safe atomic operation, however; this is not guaranteed
70 on all systems. A project should use care to ensure that their system will behave as expected. These usages are under
71 review and will be corrected over time.
72 
73 When running F´ on multi-platform systems, users typically define a deployment for each platform in the system. These
74 deployments are then linked over the platform's inter-communication architecture. Should users want F´ execution
75 across these deployments to look like a single F´ deployment, users are advised to adapt the
76 [hub pattern](../best/hub-pattern.md) to invoke F´ port calls across multiple devices.

78 ## Thread Virtualization
79 
80 **Note:** this is an experimental technology with respect to F´. Care to understand its implementation should be taken
81 before using in a production/flight context.
82 
83 Some systems, even baremetal systems, require the use of **Active Components**. Many of the `Svc` components are by
84 design active components. It is impractical to assume that all projects can, at the moment of conception, discard all
85 use of framework provided **Active Components**. Thus F´ was augmented with the ability to virtualize threading, such
86 that projects could use these components during development as they migrate to a fully passive-component system.
87 
88 To activate this feature see: [Configuring F´](../user/configuration.md). Continue reading for all the fancy details of
89 how this system is designed. TODO: put a link to the thread scheduler API documentation.

91 ### Defining Custom Tasks
92 
93 When using the thread virtualization technology, care should be taken with custom tasks/threads. This design, as
94 described below, is dependent on threads that "run once" and externalize the looping part of the thread. Therefore,
95 custom tasks must wrap functions that obey the following implementation requirements:
96 
97 1. The function shall not loop
98 2. The function shall never block execution
99 3. The function shall perform "one slice" of the thread then return
100 
101 Failure to comply with these requirements will cause the thread virtualization technology to fail, and the F´
102 application to lock-up or otherwise behave erratically.
103 
104 ### How It Works
105 
106 At the core of **Active Components** is a thread, which typically requires an Os to provide a scheduler for it to run,
107 and through this scheduler it gets designated an execution context to run in. Thus threads can execute as if they fully
108 own their execution context and the Os masks this behind the scenes. The purpose of the thread virtualization when
109 enabled for an F´ project is to unroll these threads such that they can share a single execution context and the
110 parallel behavior of the threads is "virtualized". The technique is known as protothreading. We'll explore this concept
111 with relation to F´ below.
112 
113 Each F´ thread supporting an Active component can be roughly modeled by the code below. The thread loops until the
114 system shuts down. For each iteration through the loop it blocks (pauses execution) until a message arrives. It then
115 dispatches the message and returns to a blocked state waiting for the next message.
116 
117 ```C++
118 Component1 :: run_thread() {
119  while (!shutdown) {
120  msg = block_get_message();
121  dispatch(msg);
122  }
123 }
124 ```
125 
126 Here `block_get_messages();` retrieves messages, blocking until one arrives. This loop could have easily been
127 implemented using a less-efficient model by iterating continuously through the loop and checking if a message has
128 arrived and dispatching if and only if a message is available. As can be seen below, the wait-by-blocking has been
129 replaced by the busy wait of constantly iterating through the loop.
130 
131 ```C++
132 void Component1 :: run_once() {
133  if (message_count() > 0) {
134  msg = nonblock_get_message();
135  dispatch(msg);
136  }
137  return;
138 }
139 
140 Component1 :: run_thread() {
141  while (!shutdown) {
142  comp1.run_once();
143  }
144 }
145 ```
146 
147 Here, we extracted the iteration into a `run_once` function such for clarity. The blocking wait in the first function is
148 replaced with a spin on an if-condition until a message is available, then the dispatch happens.
149 
150 It should be only a slight extrapolation that one could move all the component `run_once` functions into a single loop
151 and call each in succession. As long as these calls returns in a reasonable amount of time, and none of these calls
152 block internally, then crude parallelism is achieved.
153 
154 ```C++
155 while (!shutdown) {
156  comp1.run_once();
157  comp2.run_once();
158  comp3.run_once();
159  ...
160 }
161 ```
162 Here, as seen above, `run_once` does not block and so each component gets a slice of execution time before yielding to
163 the next. Parallelism has been virtualized and the processor is shareable without writing a full-blown thread scheduler
164 nor requiring processor instruction set support to switch threading contexts.
165 
166 Inside F´ a parallel implementation of the active component task was implemented such that it returns rather than blocks
167 on receiving messages. When `BAREMERAL_SCHEDULER` is enabled in the F´ configuration, this alternate implementation is
168 used. Under `Os/Baremetal` an implementation of a sequential scheduler is implemented. This scheduler snoops on task
169 registration and will call all thread executions in a loop driven from the main program loop similar to below.
170 
171 ```C++
172 setup(); // Setup F´
173 while (true) {
174  scheduler.run_once();
175 }
176 ```
