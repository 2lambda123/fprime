1 # Recommended F′ Development Process
2 
3 The purpose of this guide is to layout the standard F´ development process. This is the process used
4 by most developers who use F´ and, as such, many of the F´ tools are written to support the stages
5 of the process. This guild will walk through each step in this process.
6 
7 The process:
8 1. [High Level Design](#high-level-design): Draft high level requirements and architecture.
9 2. [Setup Deployment](#setup-deployment): Setup project F´ deployment
10 3. [Develop Components](#develop-ports-and-components): Create individual project components
11  - 3.1 [Design and Requirements](#design-and-requirements): Draft component requirements and interfaces
12  - 3.2 [Creating a Port](#creating-a-port): Implement component interfaces
13  - 3.3 [Creating a Component Definition](#creating-a-component-definition): Create initial mockup
14  of component
15  - 3.4 [Component Implementation](#component-Implementation): Implement component behavior
16  - 3.5 [Component Unit Testing](#component-unit-testing): Test and verify component behavior
17 4. [Assemble Topology](#assemble-topology): Compose a F´ deployment from individual components
18 5. [Integration Testing](#integration-testing): Verify high-level project behaviors
19 
20 ## High Level Design
21 
22 The first step of the development process is to establish the high-level design for the project.
23 This involves specifying system-level requirements and a block diagram that represents the key
24 system functionality. Once complete the project should break this functionality into discrete units
25 of functionality that represent the system. In addition, the interface between these units should be
26 defined. The units of functionality are Components and the interfaces are further broken down into
27 discrete call or actions through that interface. These are F´ ports. The full design of the system
28 of components and ports is the Topology. See: [Ports Components and
29 Topologies](../user/port-comp-top.md)
30 
31 Next, the project should review the components provided by the F´ framework to see what
32 functionality can be inherited for free. This usually consists of the basic command and data
33 handling components, the Os layer, drivers, and other Svc components. Where possible, these
34 components should be used as-is to support a project to minimize extra work, but these may be
35 cloned-and-owned if they fall short of project requirements.
36 
37 The project now has a list of what components they must provide, and what they will inherit.
38 
39 ## Setup Deployment
40 
41 The next step for most projects is to prepare for development. This means getting enough of a
42 deployment setup such that the developers can be assigned components and ports to implement, and
43 test within a working deployment.
44 
45 There are two options for creating a deployment:
46 
47 You can create an in-tree deployment, where a deployment is created within the F´ git repository.
48 This is convenient and requires minimal setup, but placing your code within the F´ source tree
49 can make it harder to update F´ in the future if the project wants to take advantage of future
50 bug fixes and features. To create a in-tree deployment, the Ref application can be copied and used
51 as a starting point.
52 
53 Another option is creating an standalone deployment, where the deployment is setup by itself and
54 points to a F´ installation. This requires more upfront work to setup, but provides the benefit
55 of separating mission code from F´ framework code.
56 
57 The layout for a git repository for a standalone repository might look something like:
58 
59 ```
60 mission
61 ├── mission_deployment_1
62 ├── fprime (git submodule to fprime repository)
63 ├── ...
64 └── library (git submodule to an external library)
65 ```
66 
67 After creating the project layout, an external deployments can also be created by coping the Ref
68 application.
69 
70 After copying the Ref app, both in-tree and standalone deployments need to create and modify a
71 `settings.ini` file in the root of the deployment to set the deployment toolchain, library
72 locations, and for standalone deployments, the location of the F´ framework.
73 
74 For details on configuring the `settings.ini` file, see the [user guide page](../user/settings.md).

77 ## Develop Ports and Components
78 
79 Next, each developer is typically assigned a set of components to develop. This development starts
80 with the ports that are defined to be used by the component, and then the component itself. This
81 development is described in the following sections.
82 
83 ### Design and Requirements
84 
85 Using the high level requirements, the developer should define requirements for an individual
86 component. These requirements should define the behaviors of the component, as well as the
87 interfaces with other components.
88 
89 ### Creating a Port
90 
91 Once the interfaces between components have been defined, ports should be created to implement those
92 interfaces.
93 
94 It's recommended that ports are kept in their own directories separate from components.
95 
96 To create a new port:
97 
98 1. If necessary, create new port directory
99 2. Create a new port `*Ai.xml` file, possibly by copying from an existing port.
100 3. Add the new port xml file to `SOURCE_FILES` in the `CMakeLists.txt` file in the directory
101 4. If necessary, add port directory to the deployment's cmake file with `add_fprime_subdirectory`.
102 
103 ### Creating a Component Definition
104 
105 The first step in creating a component is to create the component xml definition, which defines
106 which interfaces it implements, what commands it supports, which telemetry it provides, and what
107 events it produces.
108 
109 To create a new component definition:
110 
111 1. Create a new component directory
112 2. Create a new component `*Ai.xml` file, possibly by copying from an existing component.
113 3. Optional, create a commands xml file for GDS commands and include it in the component xml file.
114 4. Optional, create an events xml file and include it in the component xml file.
115 5. Optional, create a telemetry xml for telemetry channels and include it in the component xml file.
116 6. Create a component `CMakeLists.txt` test file and add the component xml file to `SOURCE_FILES`
117  variable in the file.
118 7. Add component directory to the deployment's cmake file with `add_fprime_subdirectory`.
119 
120 ### Component Implementation
121 
122 Next, the developer typically runs `fprime-util impl` to produce `-template` files of the hand-coded
123 .cpp and .hpp files. These can be used as a basis for implementation with all the stubs in place for
124 the developer to implement the design. Developers then fill in these files and stubs with an
125 implementation that supports the functionality of the design.
126 
127 The component can then be built as development proceeds to look for errors. Ports are entirely
128 autogenerated and do not need an implementation.
129 
130 ### Component Unit Testing
131 
132 Along with implementation, unit tests can be templated and implemented to test against the
133 requirements of the component. These should be developed an run often to ensure the implemented
134 component works.
135 
136 To add unit tests to a component:
137 
138 1. Create a test directory within the component (usually called `test`).
139 2. Run `fprime-util impl --ut` to generate unit testing code skeleton
140 3. Add unit test sources to `UT_SOURCE_FILES` and register unit tests with `register_fprime_ut()` in
141  the component `CMakeLists.txt`.
142 4. Run `fprime-util check` in the component directory to run unit tests.
143 
144 ## Assemble Topology
145 
146 As components are completed, it is helpful to add them to the topology. As more components are
147 completed, the topology is slowly built up over time. This can enable integration tests early on in
148 the project. The full deployment should be built at this stage to ensure that there are no errors.
149 
150 To add a component to the topology:
151 
152 1. In the topology `*Ai.xml` file
153  - Import the component `*Ai.xml` xml file.
154  - Instantiate the component as many time as necessary.
155  - Connect component output ports with the corresponding input port and vice versa.
156 2. In the topology `Components.hpp` file, declare the component with the same name as the topology
157  xml file.
158 3. In the topology `Topology.cpp` file:
159  - Instantiate the component.
160  - Call the component's `init` function.
161  - If additional setup is required, call a user defined setup function.
162  - If using commands, register component's commands.
163  - If using health checking, add component to ping entries.
164  - If using an active component, start component with `start` function and call `exit` when
165  exiting.
166 
167 ## Integration Testing
168 
169 As the topology comes together, it is helpful to write system level tests for subsystems of the
170 overall deployment. This makes sure that as a system, top-level requirements are met.
171 
172 The fprime-gds has a python API that can be used to write integration test cases, supporting sending
173 commands, checking for events, and getting telemetry channel readings. To get started writing
174 integration test, check out the [GDS integration test guide](../dev/testAPI/user_guide).
175 
176 ## Conclusion
177 
178 This guide was an overview of the workflow most F´ developers follow, from the start to completion
179 of a project. These workflows can be applied to any F´ project and should hopefully provide the
180 smoothest development experience.
