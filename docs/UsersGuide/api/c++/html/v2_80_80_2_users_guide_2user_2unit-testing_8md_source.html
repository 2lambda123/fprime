1 # Unit Testing in F´
2 
3 Testing is an important part of flight software (FSW) development.
4 Testing is divided into two phases: i) unit testing, and ii) integration
5 testing. Unit testing tests the individual units, such as F′ components,
6 while integration testing tests the integrated system. Test framework
7 classes include the auto-generated *TesterBase*, the auto-generated
8 *GTestBase*, and the developer-written *Tester*. The testing phases and
9 test framework classes are discussed in further detail below.
10 
11 ## Unit Testing
12 
13 Thorough unit testing is critical. It provides unit-level regression
14 tests, and makes integration easier since localized errors are caught
15 early and system-level issues only appear during integration.
16 
17 F′ provides the support for unit testing at the component level. The
18 overall framework for unit testing is shown in Figure 1.
19 
20 ![Test](../media/test1.png)
21 
22 **Figure 1.** Unit testing framework overview.
23 
24 The goal of unit testing is to cover all component-level requirements,
25 and achieve close to 100% code coverage and a reasonable amount of
26 system state and path coverage. These requirements should drive the
27 tests. A record of how the tests cover the requirements should be
28 maintained. Mapping the tests to requirements can be recorded on a
29 spreadsheet, or in the actual test using comments or in the console
30 output mechanism.
31 
32 To start, generate the test classes and add public test methods to the
33 tester. The *TesterBase* and the *GTestBase* are auto-generated, while
34 the *Tester* is developer-written from a generated template.
35 
36 ***TesterBase*** is the base class for testing a component and provides
37 a harness for unit tests. The *TesterBase* interface is the mirror image
38 of the component (*C*) under test. For each output port in *C* there is
39 an input port called a “from port,” and for each input port in *C* there
40 is an output port called a “to port.” For each “from port” there is a
41 history (*H*) of data received through a virtual input handler that
42 stores its argument into *H*. The *TesterBase* provides utility methods
43 for writing tests to the component. These include sending commands,
44 sending invocations on to ports, and getting and setting parameters and
45 time.
46 
47 ***GTestBase*** is derived from the *TesterBase* and includes headers
48 for the Google Test framework with F′ specific macros. It supports test
49 assertions, such as *ASSERT\_EQ(3, x)* to check that two values are
50 equal when writing tests. The F′ specific macros check the telemetry
51 received from the ports, the events received from the ports, and the
52 data received (user-defined) from the ports. The *GTestBase* is factored
53 into a separate class so its use is optional on systems that do not
54 support it.
55 
56 ***Tester*** is derived from the *GTestBase* and contains the component
57 under test as a member. The autocoder provides a template where the user
58 then adds tests as a public method, and also write tests in a derived
59 class of the *Tester*.
60 
61 Once the test class is generated, the user can begin to send commands,
62 check events and telemetry, check user-defined output ports, set
63 parameters and the time, build and run the unit tests from the
64 *component directory*, and finally analyze the code coverage from the
65 *component directory* after building and running the test. However, be
66 sure to review the analysis from the *test/ut directory*.
67 
68 A standard approach to writing unit tests is to write a complete test
69 that covers the requirement. If there is overlap, refactoring into
70 functions is the preferred approach to avoid code duplication. A more
71 disciplined approach would be to write functions that test individual
72 behaviors. When writing test code treat unit testing as a programming
73 problem by applying similar style guidelines as the flight code. This
74 approach avoids less code duplication and provides more readable,
75 maintainable, and modifiable tests.
76 
77 When writing unit tests for a component be sure to test against the
78 interface, such as the send commands and the send data on the output
79 ports. Read the internal component state to verify it is good, and only
80 modify the state through the interface; do not update the state of the
81 component. This approach leads to a more structured test. If there is a
82 requirement to test a function in a component implementation that has a
83 complex algorithm, then write a test against the function interface.
84 
85 When unit testing a component, model the external behavior to receive
86 commands and send responses by writing a test harness. This approach
87 supports modularity testing that can be used for many tests.
88 
89 To check event and telemetry histories the user first sends a command,
90 and then checks events and telemetry by writing the following code.
91 
92 **Sending Commands:**
93 
94 ```
95 // Send command
96 this->sendCOMMAND_NAME(
97  cmdSeq, // Command sequence number
98  arg1, // Argument 1
99  arg2 // Argument 2
100 );
101 
102 this->component.doDispatch();
103 // Assert command response
104 ASSERT_CMD_RESPONSE\_SIZE(1);
105 ASSERT_CMD_RESPONSE(
106  0, // Index in the history
107  Component::OPCODE\_COMMAND\_NAME, // Expected command opcode
108  cmdSeq, // Expected command sequence number
109  Fw::COMMAND_OK // Expected command response
110 }
111 ```
112 
113 **Checking Events:**
114 
115 ```
116 // Send command and check response
117 …
118 
119 // Assert total number of events in history
120 ASSERT_EVENTS_SIZE(1);
121 
122 // Assert number of a particular event
123 ASSERT_EVENTS_EventName_SIZE(1);
124 
125 // Assert arguments for a particular event
126 ASSERT_EVENTS_EventName(
127  0, // Index in history
128  arg1, // Expected value of argument 1
129  arg2 // Expected value of argument 2
130 );
131 ```
132 
133 **Checking Telemetry:**
134 ```
135 // Send command and check response
136 …
137 
138 // Assert total number of telemetry entries in history
139 ASSERT_TLM_SIZE(1);
140 
141 // Assert number of entries on a particular channel
142 ASSERT_TLM_ChannelName_SIZE(1);
143 
144 // Assert value for a particular entry
145 ASSERT_TLM_ChannelName(
146  0, // Index in history
147  value // Expected value
148 );
149 ```
150 To check the user-defined output ports write the following code.
151 ```
152 // Send command and check response
153 …
154 
155 // Assert total number of entries on from ports
156 ASSERT_FROM_PORT_HISTORY_SIZE(1);
157 
158 // Assert number of entries on a particular from port
159 ASSERT_from_PortName_SIZE(1);
160 
161 // Assert value for a particular entry
162 ASSERT_from_PortName(
163  0, // Index in history
164  arg1, // Expected value of argument 1
165  arg2 // Expected value of argument 2
166 );
167 ```
168 
169 To set the parameters in a test of component *C*, write the following
170 code. This call stores the argument in member variables of *TesterBase*,
171 so when *C* invokes the *ParamGet* port it receives the argument.
172 
173 ```
174 this->paramSet_ParamName(
175  value, // Parameter value
176  Fw::PARAM_VALID // Parameter status
177 );
178 ```
179 
180 Next, to set the time in a text of component *C*, write the following
181 code. *Time* is an *Fw::Time* object, so when *C* invokes the *TimeGet*
182 port it receives the value time.
183 
184 `this->setTime(time)`
185 
186 The F′ Prime build system provides targets for building and running
187 component unit tests.
188 
189 To build unit tests, go to the component directory (not the *test/ut*
190 directory) and run `fprime-util generate --ut`.
191 
192 To run unit tests, go to the component directory (not the *test/ut*
193 directory) and run `fprime-util check [parameter flags]`.
194 
195 Unit test check parameter | Description
196 ---|---
197 `--all` | Run all unit tests, combinable with `leak` or `coverage`
198 `--coverage` | Check for code coverage in unit tests
199 `--leak` | Check for memory leaks in unit tests
200 
201 For example, to run all unit tests and check for code coverage, run `fprime-util check --all --coverage`.
202 
203 ### Choosing a test library
204 
205 Components that call into libraries have two ways to write tests:
206 
207 - Link against the library in the test
208 - Link against a mock or stub library
209 
210 If you link against the library in the test, avoid linking against the
211 mock or stub library. Linking against only the test library proves that the component code works with the actual library.
212 
213 Linking against a mock or stub library makes it easier to induce
214 behaviors for testing, like injecting faults. This approach may be
215 the only option on some platforms.
216 
217 ### Code coverage
218 
219 Code coverage checks which lines were run at least once during
220 a test. Tools like *gcov* perform code coverage analysis by compiling and running the tests, then producing a report.
221 
222 Generally, code coverage checks close to 80% of lines. The remaining
223 lines are usually off-nominal behaviors that may require additional
224 effort to check by reverse reasoning from the desired behavior to
225 synthesize the inputs, or by injecting faults into the library
226 behaviors.
227 
228 Note that 100% code coverage does not check which system states were tested, nor which paths through the code were tested.
229 
230 To review code coverage analysis, go to the component directory and review the summary output *\_gcov.txt* files. Next, go to the component directory to review the coverage annotation *.hpp.gcov* and *.cpp.gcov* source files.
