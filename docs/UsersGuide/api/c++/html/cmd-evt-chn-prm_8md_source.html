1 ## Data Constructs: Commands, Events, Channels, and Parameters
2 
3 Typically, spacecraft software is controlled through commands, and monitored using a set of events and telemetry
4 channels. These are the critical data constructs supported directly by F´ and have built in autocoder support. In
5 addition, the F´ `Svc` components handel these types making it easy to define and control an F´ system through commands,
6 events, and telemetry channels.
7 
8 Parameters allow for controlling stored values that effect the operation of an F´ system. They have framework support
9 to automatically, load, store, and set these values using commands.
10 
11 **Note:** events are sometimes called EVRs and telemetry channels are sometimes called EHAs.
12

24 ## Commands
25 
26 Each **Component** defines a set of commands for operations. Unlike pipes, which are intended for component to component communication, commands are designed for user interaction with a component. Commands are defined through a series of
27 properties. Users can send commands to the F´ system and via `Svc::CmdDispatcher` these commands are dispatched to a
28 handling component to invoke some behavior. The handling component handles a command by defining a command handler function
29 to run when the command arrives.
30 
31 The properties defining commands are shown below:
32 
33 1. opcode: a numerical value uniquely representing the command. This opcode is automatically adjusted to be relative to
34 a component's base id, and thus will not collide with other commands defined by other components.
35 2. mnemonic: a text value uniquely representing the command. The component instance name is prepended to the mnemonic
36 to ensure uniqueness in the system.
37 3. arguments: a set of primitive and F´ data types supplied to the command handler to adjust its execution. These
38 arguments are sent from the ground.
39 4. synchronization "kind": (sync, async, guarded) controls which execution context the command runs in. Sync and guarded
40 commands run on the execution context of the command dispatcher. Async commands execute on the component threads and
41 can specify a priority. Guarded commands are protected from reentrancy by a mutex. These are similar to
42 [port kinds](./port-comp-top.md#a-quick-look-at-port-kinds-defined-in-a-components-usage)
43 
44 Code in the component-specific generated base class calls a function to invoke the user defined command handler. This is
45 hooked up by connecting the command registration, command dispatch, and command response ports.
46 
47 ### Command Dispatching
48 
49 When commands are defined for a component, the autocoder automatically adds ports for registering and receiving
50 commands, and reporting an execution status when finished. Each component that handles commands should be hooked up to
51 the command dispatcher connecting the registration, dispatch, and response ports in parallel.
52 
53 ![Command Dispatcher](../media/data_model1.png)
54 
55 **Figure 4. Command dispatcher.** The command dispatcher receives the raw buffer containing the command and arguments.
56 The command opcode is extracted, and a lookup table is used to find the handling component. The argument buffer is then
57 passed to the component, and the command dispatcher waits without blocking for the component to return status..
58 
59 In many projects, commands need to be sequenced in order. In order to facilitate this the framework provides
60 `Svc::CmdSequencer`. The command sequencer reads a defined sequence of commands and sends each in turn to the command
61 dispatcher to be dispatched and the command execution status is returned to the sequencer. This is an alternate path to
62 send command buffers to the command dispatcher than the external path from ground.
63 
64 ### Command Sequencing
65 
66 ![Command Sequencer](../media/data_model2.png)
67 
68 **Figure 5. Command sequence.** The command sequencer loads a sequence file from the file system, sends the command,
69  and waits for the response for each command in the sequence. A failed response terminates the sequence,
70 while a successful response moves to the next command in the sequence.

72 ## Events
73 
74 Events represent a log of activities taken by the embedded system. Events can be thought of in the same way as a program
75 execution log in that they enable the ability to trace the systems execution. Events are sent out of the system via the
76 `Svc::ActiveLogger` component and components defining commands should hook up the log port to it. If console logging is
77 desired, the text log port can be hooked up to the `Svc::PassiveConsoleTextLogger` component. Events are defined per
78 component and are typically used to capture what the component is doing. Events can be sporadic; however, should all be
79 captured for downlink. Events are defined by the following properties:
80 
81 1. id: a numeric id uniquely define this event. It is automatically offset by the component's base id to ensure global
82 uniqueness.
83 2. name: a unique text identifier for this event. This is prepended by the component instance name to remain unique.
84 3. severity: a text identifier identifying the severity of this event. Possible values are:
85  1. DIAGNOSTIC: akin to debug messages. Usually not sent to the ground.
86  2. ACTIVITY_LO: akin to fine info messages these typically come from background tasks
87  3. ACTIVITY_HI: akin to info messages these typically come from foreground tasks
88  4. WARNING_LO: less severe warning events
89  5. WARNING_HI: high-severity warning events, although the system can still function
90  6. FATAL: fatal events indicating that the system **must** reboot
91  7. COMMAND: events tracing the execution of commands
92 4. arguments: like command arguments these are primitive and complex types that represent the variable data associated
93 with the event. These are injected into the format string for a full text representation of the event.
94 5. format string: a C-style format string used to reconstruct a text version of the event.
95 
96 **Note:** the use of events to severity levels are based on the judgement of the system designer.
97 
98 Code in the component-specific generated base class provides a function to call to emit each event defined by the
99 component. This function expects an argument to be supplied for each argument defined by the event. The code generator
100 automatically adds ports or retrieving a time tag and sending events. There are two independent ports for sending
101 events:
102 
103 1. binary log output port for sending outside the system
104 2. a text log output port for on-board consoles
105 
106 ### Event Logging
107 
108 Events first acquire a time tag to represent when they occurred and then are typically sent to the `Svc::ActiveLogger`
109 component on their way to be sent down to the ground. This logger component both process the event and also recognizes
110 and begins responses for FATAL severity events.
111 
112 ![Active Logger](../media/data_model3.png)
113 
114 **Figure 6. Event log.** The component implementation calls function to generate the event. The base class retrieves the
115 time tag from the time source component. The component sends the event to the event log component, which reads it from
116 the port queue and sends it to the ground.

118 ## Channels
119 
120 Channels, also known as Telemetry Channels, or just Telemetry, represent the current reading of some portion of system
121 state. This state is either restricted to "send on change" or "send per update" even if the update is already the
122 current value. Channels are broken-up per component and are typically sampled at a set rate and dowlinked. Channels are
123 id, time, and value triples and are defined per component with the following properties:
124 
125 1. id: the unique id of the channel. This is offset by the base id of the component for global uniqueness.
126 2. name: the unique text name of the channel. This is prepended with the component name for global uniqueness.
127 3. data_type: type of the value of the channel. Can be primitive and complex types.
128 4. update: "on_change" to update only when the written value changes, and omitted to always downlink
129 
130 Code in the component-specific generated base class provides a function to call for to set the current value for a
131 channel id. This function must be supplied with a typed argument for the value. It will request the time tag internally.
132 The code generator automatically adds ports for retrieving time tag and sending channelized data.
133 
134 ### Telemetry Database
135 
136 The telemetry database acts as a double-buffered store for telemetry values. Components are free to update channels at
137 any time, however; at a set rate the current value will be read from the telemetry database and sent to the ground.
138 Components using this service should hook up the telemetry port to the telemetry database (`Svc::TlmChan`).
139 
140 ![Telemetry Database](../media/data_model4.png)
141 
142 **Figure 7. Telemetry database.** The telemetry database has a double-buffered array of telemetry buffers. The base
143 class function retrieves the time tag from the time source component, and then writes the updated value to the telemetry
144 database component. The telemetry database is called periodically to send the current set of telemetry to the ground.
145 
146 **Note:** the periodic call to the telemetry database is typically made by a [rate group](../best/rate-group.md).
147 
148 ## Parameters
149 
150 Parameters are traditional means of storing non-volatile states in the embedded system. The framework provides code
151 generation to manage parameters defined by a component. Parameters are defined by the following properties:
152 
153 1. id: unique id of the parameter, offset by component base id to ensure global uniqueness
154 2. name: unique text name of parameter prepended by component instance name for global uniqueness.
155 3. data type: primitive or complex types that represent the type of the value stored.
156 4. default value: default values are assigned in the event the parameter cannot be retrieved.
157 
158 The code generator automatically adds ports for retrieving parameters. During initialization, a public method in the
159 class is called which retrieves the parameters and stores copies locally. Calls can reoccur if the parameter is updated.
160 he code generated base class provides a function to call for each parameter to retrieve the stored copy; and an
161 implementation class can retrieve the value whenever the parameter value is needed.
162 
163 ### Parameter Database
164 
165 The framework provides the ability to store these parameters in the parameter database (`Svc::PrmDb`). This component
166 provides ports to get and set parameters, which are stored in a file to persist across reboots.
167 
168 ![Parameter Database](../media/data_model5.png)
169 
170 **Figure 8. Parameter manager.** The parameter manager or database loads the file containing parameters from the file
171 system during initialization. The initialization subsequently calls *loadParameters()* on components with parameters.
172 Components can set and retrieve parameters. The parameter manager saves the updated values to the file system via the
173 command.
