2 # Svc::FileDownlink Component
3 
4 ## 1 Introduction
5 
6 `FileDownlink` is an active F´ component that manages spacecraft file downlink. Both operators and
7 components on the spacecraft can add files to the file queue, which `FileDownlink` will downlink
8 from. Operators can enqueue files using the `SendFile` and `SendPartial` commands, and components
9 can enqueue files using the `SendFile` port. The `FileComplete` port broadcasts when a file downlink
10 initiated by a port completes, allowing components to detect when a previous enqueued file downlink
11 has completed. To prevent a continuous stream of file downlink traffic from saturating the
12 communication link, a cooldown can be configured to add a delay between the completion of a file
13 downlink and starting on the next file in the queue.
14 
15 ## 2 Requirements
16 
17 Requirement | Description | Rationale | Verification Method

23 ## 3 Design
24 
25 ### 3.1 Assumptions
26 
27 The design of `FileDownlink` assumes the following:
28 
29 1. File downlink occurs by dividing files into packets
30 of type [`Fw::FilePacket`](../../../Fw/FilePacket/docs/sdd.html).
31 
32 2. One file downlink happens at a time.
33 
34 3. Both components and operators must be able to enqueue files, necessitating both a `SendFile`
35  command and port.

60 ### 3.4 Constants
61 
62 `FileDownlink` has the following constants, initialized
63 at component instantiation time:
64 
65 * *downlinkPacketSize*: The size of the packets to use on downlink.
66 * *timeout*: Max amount of time in ms to wait for a buffer return before aborting downlink
67 * *cooldown*: The amount of time in ms to wait in a cooldown state before starting next downlink.
68 * *cycle time*: Frequency in ms of clock pulses sent to `Run` port, used for timing timeouts and
69  cooldown.
70 * *file queue depth*: The maximum number of files that can be held in the internal file downlink
71  queue. Attempting to dispatch a SendFile command or port call while the queue is full will result
72  in a busy error response.
73 
74 ### 3.5 State
75 
76 `FileDownlink` maintains a *mode* equal to
77 one of the following values:
78 
79 * IDLE (0): `FileDownlink` is idle.
80 
81 * DOWNLINK (1): `FileDownlink` is performing a file downlink.
82 
83 * CANCEL (2): `FileDownlink` is canceling a file downlink.
84 
85 * WAIT (3): `FileDownlink` is waiting for a buffer to be returned before sending another packet.
86 
87 * COOLDOWN (4): `FileDownlink` is waiting in a cooldown period before downlinking the next file.
88 
89 The initial value is IDLE.
90 
91 ### 3.6 Commands
92 
93 `FileDownlink` recognizes the commands described in the following sections.
94 
95 #### 3.6.1 SendFile/SendPartial
96 
97 SendFile is an asynchronous command that adds a file to the file downlink queue.
98 It has two arguments:
99 
100 1. *sourceFileName*: The name of the on-board file to send.
101 2. *destFileName*: The name of the destination file on the ground.
102 
103 SendPartial also includes the following fields:
104 
105 3. *offset*: Position in file to start reading from.
106 4. *length*: Amount of data to read. A length of 0 reads until the end of file.
107 
108 When the downlink completes or fails, a CmdResponse packet will be sent indicating success or
109 failure.
110 
111 #### 3.6.2 Cancel
112 
113 Cancel is a synchronous command.
114 If *mode* = DOWNLINK, it sets *mode* to CANCEL.
115 Otherwise it does nothing.
