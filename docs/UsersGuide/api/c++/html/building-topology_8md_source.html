2 
3 **Note:** for a hands-on walk through of build topologies, please see: [Tutorials](../../Tutorials/README.md)

5 The executing software consists of a set of interconnected components
6 executing on the threads of the active components or driven by other
7 events in the system such as hardware interrupts or timing sources. This
8 section will describe the steps necessary to get the software up and
9 running.

21 ## Instantiating the Components
22 
23 The constructors in the component base classes have been designed so
24 that the components can be instantiated using whatever memory model the
25 developer wishes. They can be created statically, on the heap, or on the
26 stack. As described in Section 6.7.8.1, the constructor has either a
27 name argument or none at all. The developer’s derived class constructors
28 may have extra arguments that are particular to that application. If
29 classes are declared statically, the developer should keep in mind
30 uncertainties about execution ordering.
31 
32 ## Initializing the Components
33 
34 As discussed in Section 6.8.2, each component has an init() call that
35 initializes the component base classes. This call should be made after
36 instantiating the components. For queued and active components, the
37 queue size is passed. The queue should be sized based on an
38 understanding of task priorities and message traffic between the
39 components.

41 ## Interconnecting the Components
42 
43 The components in the software are interconnected by connecting the
44 ports of the components together. Ports are connected by passing
45 pointers to input ports to the output ports that are calling them.
46 Methods are generated in the component base classes to get input port
47 pointers and pass them to output ports. The following sections describe
48 the connections of different port types.
49 
50 ### Interface Ports
51 
52 Interface ports are the regular ports that are used to connect
53 components together. For each port type and name on a component, the
54 method naming scheme is as follows:
55 
56 Get input port pointer:
57 
58 > <PortType>\* get\_<port name>\_InputPort(NATIVE\_INT\_TYPE
59 > portNum);
60 
61 where
62 
63 > <PortType> = The full port type specified in the data\_type
64 > attribute in the definition of the port in the component XML.
65 >
66 > <port name> = The name of the port in the name attribute of the port
67 > definition.
68 
69 The portNum argument to the method should be set to zero if there is
70 only one instance of the port.
71 
72 Set output port pointer:
73 
74 The value of the input pointer retrieved via the method in the last
75 section is given to an output port of the same type by calling:
76 
77 > void set\_<port name>\_OutputPort(NATIVE\_INT\_TYPE portNum,
78 > <PortType>\*port);
79 
80 where
81 
82 > <PortType> = The full port type specified in the data\_type
83 > attribute in the definition of the port in the component XML.
84 >
85 > <port name> = The name of the port in the name attribute of the port
86 > definition.
87 
88 The portNum argument to the method should be set to zero if there is
89 only one instance of the port.
90 
91 There is a second overloaded version of the method to set an output port
92 when the input port being passed to it is a serialized port:
93 
94 > void set\_<port name>\_OutputPort(NATIVE\_INT\_TYPE portNum,
95 > Fw::InputSerializePort \*port);

97 ### Command Ports
98 
99 As discussed in Section 6.6.4, the code generator will create the
100 correct set of command-related ports for a component that has commands
101 defined. For that component, the functions used to get or set
102 command-related port pointers have standard names. The names are as
103 follows:
104 
105 Get command input port:
106 
107 > Fw::InputFwCmdPort\* get\_CmdDisp\_InputPort(void);
108 
109 Set command status port:
110 
111 > void set\_CmdStatus\_OutputPort(Fw::InputCmdResponse\_Port\* port);
112 
113 Set command registration port:
114 
115 > void set\_CmdReg\_OutputPort(Fw::InputCmdRegPort\* port);
116 
117 For the component(s) that are connected to those ports, they would use
118 the normal methods for accessing the port pointers as described in the
119 last section.
120 
121 ### Telemetry Ports
122 
123 The standard port accessor functions for telemetry are as follows:
124 
125 Set telemetry output ports:
126 
127 > void set\_Tlm\_OutputPort(Fw::InputTlmPort\* port);
128 
129 Set time output ports:
130 
131 > void set\_Time\_OutputPort(Fw::InputTimePort\* port);
132 
133 ### Event Logging Ports
134 
135 The standard port accessor functions for logging events are as follows:
136 
137 Set logging output ports:
138 
139 > void set\_Log\_OutputPort(Fw::InputLogPort\* port);
140 >
141 > void set\_TextLog\_OutputPort(Fw::InputFwLogTextPort\* port);
142 
143 Set time output ports:
144 
145 > void set\_Time\_OutputPort(Fw::InputFwTimePort\* port);
146 
147 Note that the set\_Time\_OutputPort() call is shared with the telemetry
148 ports. It can be called once for both.

154 > void set\_ParamGet\_OutputPort(Fw::InputFwPrmGetPort\* port);

158 ## Registering Commands
159 
160 The pattern for dispatching commands is for a user-implemented command
161 dispatch component to connect an output command port to the input
162 command port for each component servicing commands. Internally, the
163 dispatcher would map the set of opcodes for a particular component to
164 the port that is connected to that component. To aid that process, the
165 code generator creates a command registration function when there are
166 commands specified for a component. It takes as an argument the port
167 number on the dispatcher component that is connected to the component’s
168 command port. The registration port should be connected to the
169 dispatcher’s registration port as described in Section 6.8.3.2. Then the
170 regCommands() method can be called in the component, which will invoke
171 the registration port for each of the opcodes defined. This method
172 should be called for each component that has commands.
173 
174 ## Loading Parameters
175 
176 Section 6.8.3.5 describes how to connect a parameter output port to a
177 component providing parameter storage. After the two components are
178 connected, the base class method loadParameters() can be called. That
179 method will request the values of all parameters for that component via
180 the parameter port. From then on, they will be available to the
181 implementation class for use. Although it is common to only read
182 parameters at software initialization, there is nothing that prevents a
183 re-read after the software has started running by invoking the
184 loadParameters() call again in the event that the parameter storage was
185 updated.

187 ## Starting Active Components
188 
189 The last action in constructing the topology is to start the tasks for
190 any active components. The start() method is found in the
191 ActiveComponentBase base class in Fw/Comp/FwActiveComponentBase.hpp.
192 Table 22 provides the arguments and their meanings.
193 
194 **Table 22.** Active component start()
195 arguments.
196 
197 | Argument | Meaning |
198 | ---------- | ------------------------------------------------------------------------------------------------------------- |
199 | identifier | A thread-independent value that is used to identify activities of the thread. Should be unique in the system. |
200 | priority | The execution priority of the task: 0 = low priority, 255 = high priority. |
201 | stackSize | The size of the stack given to the task. |
202 
203 As mentioned in Section 6.7.8.3, the functions preamble() and
204 finalizer() will be run once before and after the loop waiting for port
205 invocations.
