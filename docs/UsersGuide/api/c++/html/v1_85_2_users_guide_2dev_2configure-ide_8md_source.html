14 ## Basic IDE Setup: CMake and Variables
15 
16 F´ typically builds by-hand using the F´ utility. This utility automates away many of the details of setting up an F´
17 build and should make it easy for users to build, run, and test F´ from the commandline. If this approach works for
18 your project, we recommend you use it. However, if you need IDE support these instructions are for you.
19 
20 This guide will walk you through the setup and build of an F´ application through your IDE's CMake plugin. For helper
21 tasks like generating templated files, it is probably best to use just to continue using `fprime-util`, but for coding
22 tasks like building, running UTs, generating indices for the IDE's helper functions, debugging, etc, we will attempt to
23 setup the IDE correctly.
24 
25 **Note:** although it may be possible to integrate your IDE directly with the `fprime-util` build caches, we will not
26 attempt this here as it may result in many errors.
27 
28 ### Step 1: CMake Support for Our Beloved IDE
29 
30 F´ is built using CMake and thus to integrate with your IDE, you project must be able to support CMake builds. Some IDEs
31 support this natively, and some require plugins. Apart from the examples below, you should understand in general how
32 your beloved IDE integrates with CMake builds and how to configure CMake cache variables like setting a toolchain file.
33 
34 ### Step 2: Identifying F´ Build Information
35 
36 The CMake requires that several build settings are set. The `fprime-util` sets these automatically from the
37 `settings.ini` file, however; here we will need to configure this by hand as we are using CMake directly.
38 
39 Typically a user wants to build a `Testing` build for the default toolchain using an internal-to-F´ branch. If you are
40 just learning F´, use these settings to run and test on your personal computer hardware and skip to Step 3.
41 
42 First the user should determine whither they would like a `Testing`, `Release`, or `Debug` build. The user should
43 typically choose `Testing`, which allows for running unittests, enables debug flags, and links in all of the test code.
44 `Release` and `Debug` builds are provided by default by CMake but are used by F´ only to create the executable and not
45 the test setup.
46 
47 Next, the user should choose a toolchain file. If you are building for your host system to debug or test a native
48 executable just choose the default. If you must run on a target platform and want to build for it, choose the toolchain
49 file you would use to build for that platform.
50 
51 Third, the user should determine if they are using an internal-to-F´ build, or an external-to-F´ build. If your code is
52 placed within the F´ framework checkout (code supplied by F´ team), you are using the former. If your code specifies
53 where to find F´ code and lives outside the F´ framework directory you are suing the latter.
54 
55 Finally, identify any other build options or cache variables you need. These are anything you supply to the build int
56 the form of -D options. This can include custom configuration setups.
57 
58 ### Step 3: Setting Up CMake and Build Variables
59 
60 Through the setup of the IDE you should now set the following variables. Note, some of these settings are exposed by
61 the IDE as deliberate settings, but they can be set as part of the build setup.
62 
63 | Variable | Value | Typical |
64 |----------|-------|---------|
65 | -DCMAKE_BUILD_TYPE | Testing, Debug, or Release | Testing |
66 | | | |
67 | **Custom Toolchain** | | |
68 | -DCMAKE_TOOLCHAIN_FILE | path/to/toolchain/file | **unneeded** |
69 | | | |
70 | **Out of Source Builds** | |
71 | -DFPRIME_FRAMEWORK_PATH | path/to/fprime/framework | **unneeded** |
72 | -DFPRIME_PROJECT_ROOT | path/to/project/root | **unneeded** |
73 | -DFPRIME_LIBRARY_LOCATIONS | path/to/lib:path/to/lib:... | **unneeded** |
74 | | | |
75 | **Custom Configuration** | | |
76 | -DFPRIME_AC_CONSTANTS_FILE | path/to/project/ac/file | **unneeded** |
77 | -DFPRIME_CONFIG_DIR | path/to/project/conf/dir | **unneeded** |
78 
79 **Note:** See [CMake Settings](cmake-settings.md) for a further description of these and other settings.

81 ## IDE Examples
82 
83 IDEs change all the time and configuring them is not something a project like F´ should take on as a documentation task
84 as the overhead to support this would be immense. However, in so much as IDEs are used by the F´ team and community, we
85 will provide this section to give users and developers the most help possible even if it is not complete.
86 
87 **Note:** ensure that you understand the information in [basic IDE setup](#basic-ide-setup-cmake-and-variables). When
88 IDEs change, it is typically not too-hard to configure them from an old example and an understanding of what settings
89 are needed.
