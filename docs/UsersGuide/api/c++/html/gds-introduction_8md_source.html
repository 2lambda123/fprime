1 # A Brief Guide to the F´ Ground Data System
2 
3 The F´ Ground Data System (GDS) was built to aid in the usage, development, test, and integration of F´-based embedded
4 systems. It was designed for projects that have not already chosen a GDS, for developers who need to test before the
5 project's GDS is fully online, and for integration testers who wish to automate tests against F´ software. This guide
6 will give you a quick introduction to what the F´ GDS is and how you can use it.

29 ## What is the GDS?
30 
31 A GDS is the "Ground Data System" that supports large-scale embedded systems by allowing operators to interact with the
32 embedded system. This system is typically used with spacecraft to facilitate control and monitoring of those systems'
33 flight software. Any data being sent from a ground computer to the spacecraft, or vice versa, has to passes through a
34 GDS.
35 
36 The F´ GDS provides this capability out-of-the-box to developers and projects using F´. This means that functions
37 traditionally performed through the GDS like systems testing, operation testing, and developer debugging can happen
38 as soon as F´ is installed. This frees users of F´ to quickly get started on processes that usually occur at the end of
39 the project even if a different GDS will eventually be used.
40 
41 The rest of this guide will talk about how to use the built-in F´ GDS. Let's take a look!
42 
43 **Note:** the `Ref` application will be used as the flight software for exploring the GDS. It will work with your
44 project too!
45 
46 ## Getting Started
47 
48 The F´ GDS is a minimal amount of Python code and JavaScript that is automatically installed when you run the F´
49 installation instructions in the [installation guide](../../INSTALL.md). This guide will walk the user through running
50 the GDS, setting options, and letting the user loose to run the system.
51 
52 
53 ### Running the GDS
54 
55 To start the GDS with the default settings (IP adapter, default ports, loading the default dictionary for the project),
56 navigate to the project directory and run `fprime-gds`. This will launch both the `Ref` flight software, the GDS layer,
57 and launches the browser pointed at the GDS.
58 
59 ```bash
60 $ cd fprime/Ref
61 $ fprime-gds
62 ```
63 
64 **Note:** ensure the virtual environment has been activated. Consult the installation guide for more information.
65 
66 `Ref` is just an example application that comes with F´; you can run `fprime-gds` in any deployment project folder. For
67 this to work properly, the user must build and install the `Ref` application using `fprime-util`.
68 
69 By default, the `fprime-gds` command uses the current deployment to find the dictionary and the flight software to run.
70 The dictionary is used for the GDS to understand the events, channels, and commands produced by the embedded system and
71 the software is run as a convienence for local testing (`-n` will disable this feature).
72 
73 In the following sections, we will explore various options for using the GDS.
74 
75 ## GDS Options
76 
77 While fine for basic testing, you may want to change the operation of the GDS to meet project needs. Here are a few
78 common scenarios where most users will want to modify operation.
79 
80 To learn the F´ GDS UI, [skip to this section](#navigating-the-gds-gui).
81 
82 ### Specify the Deployment or Dictionary
83 
84 By default, the GDS will try to use the current working directory to find the deployment or dictionary it needs to run.
85 If a user wants to specify the deployment, specify the `-d DEPLOY_DIR` option. This can be run from anywhere and does
86 not require `cd`.
87 
88 ```bash
89 $ fprime-gds -d path/to/deployment
90 ```
91 
92 The GDS does not need to automatically run flight software (see next section) and thus doesn't strictly need a full
93 deployment directory. The user can simply specify the `--dictionary` for basic operations. **Note:** the GDS logs will
94 be placed in the user's home director when the `-d`/`--deploy` flag isn't set.
95 
96 ```bash
97 $ fprime-gds -n --dictionary path/to/dictionary/xml
98 ```
99 
100 **Note:** `-n` is described below.
101 
102 ### Disable Automatic Flight Software Execution
103 
104 By default, starting the GUI will also execute any native build of the deployments' F´ flight software instance. Often,
105 this becomes unnecessary as testing migrates to the embedded system. This automatic execution can be disabled by
106 supplying the `-n` flag. This also means the GDS is not required to be supplied a `-d`/`--deploy` flag, and the user can
107 run it with just the `--dictionary` and `-l`/`--log` flags.
108 
109 ```bash
110 $ fprime-gds -n
111 ```
112 
113 ### Specify GDS Addresses and Ports
114 
115 By default, the GDS runs on the local machine at address `0.0.0.0` and uses the following local ports:
116 
117 - 5000: hosts the HTML frontend. The browser connects here.
118 - 50000: the default communication interface is a port. The embedded system connects here.
119 - 50050: an internal port used for hosting GDS data.
120 
121 If a user wishes to change the IP address and port used to allow the embedded system connection to the GDS, the user
122 should specify the `--ip-address ADDRESS` and `--ip-port PORT` flag. **Note:** these options are only available when
123 using the default IP connector.
124 
125 ```bash
126 $ fprime-gds --ip-address 8.8.8.8 --ip-port 12345
127 ```
128 
129 If you need to change the IP or port of the internal server, you can do so by passing the `--tts-addr IP_ADDRESS` and
130 `--tts-port TTS_PORT` options, respectively. This allows the user to avoid collisions with other ports.
131 
132 ```bash
133 $ fprime-gds --tts-addr 8.8.8.8 --tts-port 12345
134 ```
135 
136 Changing the port used by the UI is done using the flask configuration by setting the `ADDRESS` and `PORT` settings in
137 the `gds.ini` file. This file can be set with the `-c` flag. The `ADDRESS` or `PORT` environment variables can also be
138 set to modify these variables.
139 
140 ### Run the GDS Without a UI
141 
142 In some cases, the user may wish to run the GDS without the HTML/browser UI loading. This can be done using the
143 `-g none` flag. This is usually done when running automatic integration testing, or using the [`fprime-cli`](gds-cli.md)
144 tool to operate without a UI.
145 
146 ```bash
147 $ fprime-gds -g none
148 ```

193 ## Navigating the GDS GUI
194 
195 When running the GDS using the default UI, the GDS should launch a browser tab that looks similar to the following. In
196 general, the available tabs are listed across the top and each view can be selected by clicking on those tabs.
197 
198 ![Launched Browser Window](../media/gds_gui_commanding.png)
199 
200 Across the top of the screen is a series of tabs: "Commanding", "Events", etc. Each of those tabs represent a piece of
201 the GDS's functionality. Each view opens when when you click on it. Next, we'll go through each tab's functions in more
202 detail in upcoming sections.
203 
204 To the right of those tabs is a "New Window" button. If you want to open a new GUI window to view multiple tabs at once
205 click that button. This new window will connect to the same backend as the first window.
206 
207 At the very top-right, there should be a green circle or a red X representing the GUI's connection status. A green
208 circle means that data is still flowing from the embedded system, while a red X means the embedded system is currently
209 disconnected and not sending/receiving any data. This is reset to an X via timeout, so you may need to configure that
210 timeout in the `config.js` file. This widget is fondly refered to as "the orb" as it quickly shows if there is data
211 flow from the embedded F´ system.
212 
213 There's also a NASA logo on the far left. It doesn't do anything right now, but hopefully the meatball continues to
214 inspires our users. This logo is also configurable in `config.js`.
215 
216 The tabs across the top perform most of the functionality of the GDS. What do the tabs contain? Do they contain things?
217 Let's find out!
218 
219 ### Commanding
220 
221 The commanding tab contains the items needed to send commands to the embedded system. All available commands are listed
222 in the "Mnemonic" dropdown box, in the form `<COMPONENT>.<COMMAND>`. The commands are found in the dictionary supplied
223 to the GDS. The user may select a command from the dropdown or type to filter the available commands. If the selected
224 command requires any arguments, they'll appear below the drop down where the values may be entered. Invalid argument
225 inputs will be outlined in red. When clicked, the "Send Command" button will transmit the selected command and input
226 arguments through the GDS and to the embedded system, while "Clear Arguments" will reset the arguments inputs
227 to their default values. Should an error occur in the GDS, it will be shown below the inputs. The commanding tab is
228 shown below. See: [Commands](../user/cmd-evt-chn-prm.md#commands)
229 
230 ![Commanding Tab](../media/gds_gui_commanding.png)
231 
232 
233 Below the command input is the "Command History" table. This records all the commands sent with their associated
234 arguments. This is recorded by the GDS and thus may include commands that did not transmit all the way to the embedded
235 system in case of communication error or other event. To search for commands, you can type something into the "Filters"
236 box and type enter, which displays only matching results. You can sort the rows by one of the table headers
237 (e.g. "Command Time") by clicking on that header. Double-clicking a historical command will re-populate the command
238 sending component and is useful to retry transmission of a command.
239 
240 ### Events
241 
242 The Events tab will display a table of all the "Events" that have happened on the embedded system and been received by
243 the GDS. See: [Events](../user/cmd-evt-chn-prm.md#events). The Events tab is shown below.
244 
245 ![Events Tab](../media/gds_gui_events.png)
246 
247 Each event is color-coded based on its "Event Severity;" there are 7 different kinds of severities:
248 
249 | Severity | Color | Description |
250 |---|---|---|
251 | DIAGNOSTIC | | Debug events not typically sent to the GDS |
252 | COMMAND | GREEN | Events produced by the command dispatcher to aid in tracing actual command execution |
253 | ACTIVITY_LO | GRAY | Low priority informational events typically tracking background process actions |
254 | ACTIVITY_HI | BLUE | High priority informational events typically tracking ground-commanded foreground actions |
255 | WARNING_LO | YELLOW | Low priority non-critical warning events |
256 | WARNING_HI | ORANGE | High priority critical warning events |
257 | FATAL | RED | Critical failure event typically resulting in embedded system restart |
258 
259 Just like the "Command History" table, these items can all be filtered and sorted based on the column headers.
260 Additionally, a "Clear" button will clear the table thus making the monitoring of new events easier. There is special
261 support for COMMAND event types, which will replace the opcode with the command mnemonic to help the user. The opcode
262 can be found by hovering over the the mnemonic and a tooltip will appear.
263 
264 ### Channels
265 
266 The Channels tab displays an updating table of the latest "Channels", or telemetry data, the GDS has received from the
267 embedded system. These represent the latest values and, by default, only telemetry channels that have been received
268 by the GDS are shown. Channels with no received value are not displayed, but can be configured (described below). Only
269 the most recently received value for each channel is shown. This view can be seen below.
270 
271 ![Channel GUIs](../media/gds_gui_channels.png)
272 
273 If you want to view a full list of all available channels, set a view to watch specific channels, or monitor all
274 channels even those which have not arrived, you can click on the "Edit View". This will show all available channels
275 and allows users to tick the checkbox next to a channels that should be shown. Any unchecked channels will be hidden.
276 To apply the view, click the "Done" . Import and Export allow downloading and uploading text files that represent these
277 views such that they may be saved. The "Edit View" is shown below.
278 
279 
280 ![Channel Edit GUIs](../media/gds_gui_channels_edit.png)
281 
282 
283 Just like the other tables, you can sort or filter these channel items.
284 
285 ### Uplink
286 
287 The uplink tab allows users to upload files to the embedded system. This is dependent on the usage of the FileUplink
288 components and a file system implementation. The uplink process has two steps: staging and uplink. First the user
289 browses for files to uplink to the system. This submits the files to a working set and allows them to curate that set
290 before committing to a uplink. Finally the user should press the "Submit Uplink" button to upload the files to the GDS,
291 add them to the outgoing queue, and start the uplink. Uplink progress can be monitored and the queue can be paused to
292 temporarily stop the uplink. Files are limited to no more than 32Mb.
293 
294 ![Uplink Tab](../media/gds_gui_uplink.png)
295 
296 ### Downlink
297 
298 The downlink tab monitors the downlink of files into the GDS. Any files that have been downlinked using the
299 `fileDownlink.FileDownlink_SendFile` command will be tracked in this tab once the packets arrive in the F´ GDS. The
300 progress of this download is tracked, and once the file has been downlinked, the user has the ability to Download the
301 files.
302 
303 ![Downlink](../media/gds_gui_downlink.png)
304 
305 ### Logs
306 
307 The logs tab allows the user to monitor logs produced on-disk by the GDS. This is a convenience to allow the user to
308 see those without going to the GDS server's ground system. Select a log from the list to see its contents, which update
309 in realtime.
310 
311 ![Logs](../media/gds_gui_logs.png)
312 
313 There are several standard logs that appear:
314 - `ThreadedTCP.log`: log from the GDS middleware server linking comm to the GDS actual
315 - `channel.log`: log of all channels received by the GDS
316 - `command.log`: log of all commands sent by the GDS
317 - `event.log`: log of all events received by the GDS
318 
319 In addition, if the GDS automatically ran flight software, its log will appear here. Any downlinking files will also
320 produce logs available to this tab.
