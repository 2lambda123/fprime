2 # Drv::TcpServer Tcp Server Component
3 
4 The TCP server component bridges the byte stream driver model interface to a remote TCP client to which this tcp server
5 connects and sends/receives bytes. It implements the callback formation (shown below) using a thread to receive data
6 and producing the callback port call. Since it is a server, it must startup and listen for client connections. Designed
7 for single client communication, it does not permit a queue of connecting clients.
8 
9 For more information on the supporting TCP implementation see: Drv::TcpServerSocket.
10 For more information on the ByteStreamModelDriver see: Drv::ByteStreamDriverModel.
11 
12 ## Design
13 
14 The manager component (typically the ground interface) initiates the transfer of send data by calling the "send" port.
15 The caller will provide a `Fw::Buffer` containing the data to send and the port call will return a status of that send.
16 These responses are an enumeration whose values are described in the following table:
17 
18 | Value | Description |
19 |---|---|
20 | Drv::SEND_OK | Send functioned normally. |
21 | Drv::SEND_RETRY | Send should be retried, but a subsequent send should return SEND_OK. |
22 | Drv::SEND_ERROR | Send produced an error, future sends likely to fail. |
23 
24 This data is immediately sent out to the remote tcp server with a configured send timeout. See Usage described below.
25 
26 **Callback Formation**
27 
28 ![Callback](../../ByteStreamDriverModel/docs/img/canvas-callback.png)
29 
30 In the callback formation, the byte stream driver component initiates the transfer of received data by calling the
31 "readCallback" output port. This port transfers any read data in a `Fw::Buffer` along with a status for the receive.
32 This status is an enumeration whose values are described in the following table:
33 
34 | Value | Description |
35 |---|---|
36 | Drv::RECV_OK | Receive functioned normally buffer contains valid data. |
37 | Drv::RECV_ERROR | Receive produced an error and buffer contains no valid data. |
38 
39 ## Usage
40 
41 The Drv::TcpServerComponentImpl must be configured with the address of the remote connection, and the socket must be
42 open to begin. Usually, the user runs the Drv::TcpServerComponentImpl engaging its read thread, which will automatically
43 open the connection. The component is passive and has no commands meaning users should `init`, `configure`, and
44 `startSocketTask`. In addition to these methods shared with the Drv::TcpClientComponentImpl, the server provides
45 `startup` and `shutdown` methods to start and stop the listening socket. It `startup` must be run before the read task
46 is started and `shutdown` should be called before the task is stopped.
47 
48 Upon shutdown, the `stopSocketThread` and `joinSocketThread` methods should be called to ensure
49 proper resource deallocation. This typical usage is shown in the C++ snippet below.
50 
51 
52 ```c++
53 Drv::TcpServerComponentImpl comm = Drv::TcpServerComponentImpl("TCP Server");
54 
55 bool constructApp(bool dump, U32 port_number, char* hostname) {
56  ...
57  comm.init(0);
58  ...
59  if (hostname != NULL && port_number != 0) {
60  Fw::EightyCharString name("ReceiveTask");
61  comm.configure(hostname, port_number);
62  comm.startup();
63  comm.startSocketTask(name, TASK_PRIORITY, TASK_STACK_SIZE);
64  }
65 }
66 
67 void exitTasks(void) {
68  ...
69  comm.shutdown();
70  comm.stopSocketTask();
71  (void) comm.joinSocketTask(NULL);
72 }
73 ```
