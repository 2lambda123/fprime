11 ## Ports: F´ Communication
12 
13 Although F´ is primarily decomposed into a set of **Components**, it is often the port that a user of F´ must
14 understand first. This is because the set of ports a **Component** defines is the external (or public) interface of
15 that **Component** in an F´ system. Thus it is essential to understand and design these **Ports** correctly.
16 
17 ### Port Characteristics
18 
19 Ports are the point of interconnection between **Components** in the software framework and encapsulate typed interfaces
20 in the architecture. This means that each port definition is of a specific type (e.g. CommandDispatchPort) and can only
21 connect **Components** defining ports of the same type. When one **Component** uses a port to communicate with another
22 **Component** this action is called a port **invocation**. These port types are known as "data_types" because they
23 represent the data being conveyed across the port.
24 
25 A port supports passing arguments across the port's connection in order to pass data to the receiving
26 **Component** of an invocation. Some ports may also return data from the receiving **Component** to the invoking
27 **Component**. A port may specify zero or more arguments of any F´ data type or primitive (int, float, U8, etc.).
28 Pointers and references are allowed as arguments to the port as well (for performance purposes), but care should be
29 taken to ensure correct memory management as the ownership of the underlying memory is effectively shared when the
30 port is invoked. The port's type is synonymous with the port invocations' "data_type" when used on a **Component**.
31 
32 When using a port, the user specifies directionality. It can represent an input, or an output. This direction
33 pertains to the direction of invocation from one originating **Component** to another and not necessarily the direction
34 of data flow i.e. output ports (invoking out to another **Component**) can in some cases retrieve data. Multiple output
35 ports can be connected to a single input port implying that at **Component** responding to a port invocation could be
36 responding to one of a set of invoking **Components**. **Caution:** a single output port can be connected to only one
37 input port at a time.
38 
39 Another aspect of port usage is the synchronous/asynchronous nature of the port. A synchronous port operates like a
40 function call running in the execution context of the invoking **Component** (i.e. on the invoker's Thread). The
41 synchronous nature of the port along with the directionality is combined in the "kind" when used by the **Component**.
42 
43 Finally, a port can be defined as "guarded", which means invocations into the port are limited to a single invocation
44 at a time by a component-wide mutex. This is also combined into the "kind" attribute.
45 
46 ![Port Image](../media/core1.png)
47 
48 **Figure 1. Port connectivity.** Ports are connected to ports of the same type. When used by a *component*, the ports
49 can define directionality. Using *serialized ports* (see blow), serializes the call and passes it to a data buffer
50 such that it can be sent and deserialized anywhere.
51 
52 When discussing ports, one must be careful to distinguish between the port **design** and the port instantiation. A
53 port's design is generic specifying its type (name, arguments transported, etc) and when used in a **Component** is
54 designated as the port's "data_type". The design is careful not to specify its usage by a component (directionality,
55 synchronous/asynchronous nature, guarded/not guarded). This is specified as part of the instantiation of a port when
56 used by a component. This instantiation is specifically part of the **Component**'s design and not the port's
57 design and is known as the "kind" of port's instantiation.

80 ### Port Call Serialization
81 
82 Serialization on ports take arguments supplied to the port and translates them to a data buffer. The serialized ports
83 are supported by the framework allowing users to send primitive and custom types. Ports can have return valves; however,
84 use is limited to only return data when the component has defined the input port (invoked port) as a synchronous
85 interface, and the port serialization has been disabled since serialization passes a data buffer without returning one.
86 
87 Serialization takes a specific set of typed values or function arguments and converts them in an
88 architecture-independent way translating them to a data buffer. A port calls’ commands and arguments are serialized and
89 placed on message queues in the receiving component. In addition command arguments and telemetry values are passed as
90 serialized buffers so that components that transfer the data can be independent of the definition of the data.
91 
92 Users can define arbitrary interface argument types and the framework automatically serializes the data for port calls.
93 They can also define complex types in XML where a code generator generates classes that are serializable for use
94 internally, and usable with the F´ supplied ground system.
95

97 ### Special Serialized Ports
98 
99 Serialization ports are special ports that handle serialized buffers without automatically unpacking them. Any output
100 port type can connect to a serialization port input type, and any serialization output port can be connected to any
101 input port type. This allows strongly-typed ports to connect to generic "pass-any-data" serialization ports as shown
102 in Figure 3. For input ports, the calling port detects a connection and serializes arguments. For the output ports,
103 the serialized port calls an interface on the typed port that deserializes arguments. Serialization ports do not support
104 ports with return types. These ports allow serialized data to be passed around by generic components that do not know
105 the type contained. The [Hub pattern](../best/hub-pattern.md) often uses serialized ports such that data can be routed
106 across a address-space gap in a generic fashion.
107 
108 ![Serialization Ports](../media/core3.png)
109 
110 **Figure 3. Serialization ports.** Input ports input the serialized buffer, while the output ports output
111 the serialized buffer.
112 
113 Serialization ports are useful for generic storage and communication components that do not need to know the type of
114 data passing through them. This allows design and implementation of command and data handling (C&DH) components that
115 provide reusability. Tested C&DH components are developed to implement typical non-mission-specific flight functions
116 
117 See note about usage for [Commands, Events and Telemetry](./cmd-evt-chn-prm.md).

120 ## Components: F´ Modules
121 
122 The F′ architecture is based on decomposing the system into modules called components. Each component contains a
123 discrete portion of the system's logic (Figure 10); The component architecture implies usage patterns, as well as
124 usage constraints.
125 
126 ![Component Architecture](../media/core10.png)
127 
128 **Figure 10. Example of F′ component architecture pattern.**
129 
130 Components encapsulate behavior, and are not aware of other components. They are localized to one compute context with
131 specific interactions with other components using ports. There should be no non-port communication between components.
132 Components are responsible for handling the invocations of ports used in the component. They may also define and handle
133 commands as well as emit telemetry and events.
134 
135 ![Example Component](../media/core11.png)
136 
137 **Figure 11. Encapsulation of Behavior** The component handles the port behaviors as well as executes the commands and
138 then produces the telemetry. It does not directly interact with other components.
139 
140 ### Component Types
141 
142 There are three types of components: passive, queued, and active. Theses components allow for different functionality in
143 the system and may use different port input types [see below](#port-kinds-available-to-each-component-type). Each type
144 of component is defined below:
145 
146 **Passive component:** has no thread and cannot support asynchronous port invocations nor asynchronous commands. Port
147 invocations call into the developer class but execution context is supplied from the invoking component.
148 
149 **Active component:** has a thread of execution as well as queue. The thread dispatches port calls from the queue
150 as on the execution context of the thread. Active components may define use port kinds. **Caution:** synchronous and
151 guarded port invocations still execute in the execution context of the invoker.
152 
153 **Queued component:** has no thread but does have a queue. Thus it handles asynchronous commands and port invocations;
154 however, the user must implement at least on synchronous port invocation that unloads and handles the messages on the
155 queue. For this and any other synchronously invocation execution context is supplied by the invoker. **Note:** this
156 component type is only rarely used. Ensure it is the correct choice for your design.
157

166 ### Division of Component Implementation
167 
168 Each component is divided into three classes that each represent a piece of the component's implement. These three
169 classes are as follows:
170 
171 1. Core Framework Class: the base-class of components defined as part of the framework. A component may inherit from:
172 active, passive, and queued classes. These represent the component types defined above.
173 2. Generated Component-Specific Base: this class is the direct descendant of the core framework class and is
174 automatically generated to provide all the implementation for framework features.
175 3. Component-Specific Developer Implementation Class: this class inherits from the generated component-specific base
176 class and contains only the user specific implementation for the component.
177

185 ### Port Kinds Available to Each Component Type
186 
187 Passive components support synchronous and guarded ports with the execution context supplied by the calling component.
188 There is no queue to support asynchronous ports. Active components support all three port types; however, the active
189 component needs at least one asynchronous port otherwise it would be effectively passive with a superfluous queue and
190 thread.
191 
192 Queued components also support all three port types; however, the queued component needs at least one synchronous or
193 guarded port as the code and one asynchronous port. A synchronous or guarded port must be defined to unload the internal
194 queue as a queued component does not have a thread to automatically unload the queue. A queued component must define one
195 asynchronous port otherwise it too would effectively be passive components with an unused queue attached.
196 
197 Output ports are invoked by calling generated base class functions from the implementation class. The behavior invoked
198 is defined input port side of the connection and thus there are no special restrictions to use output ports.
199

211 ## Topology: F´ Application
212 
213 Components are instantiated at runtime and then connected through the ports to other components in the system. This
214 graph of interconnected components is known as a topology, as shown in Figure 13. The topology graph is still designed
215 before runtime, but the actual connections of ports happens during the construction and setup phase of runtime F´
216 software. There should be no code dependencies between the components, only dependencies on port interface types. A
217 components ability to communicate with other components is enabled through the interconnections specified in the
218 topology. Alternate implementations can therefore easily be swapped, for example with simulation versions.
219

222 **Figure 13. Example of a topology.**

224 ## Conclusion
225 
226 F´ is decomposed into a federation of discrete components organized into a system graph. The interconnections in this
227 graph are handled through port calls such that the otherwise independent components can interact with each other.
