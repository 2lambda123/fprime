2 # Svc::GenericHub Generic Hub Component
3 
4 The Generic Hub component is an implementation of the F´ [hub pattern](https://nasa.github.io/fprime/UsersGuide/best/hub-pattern.html).
5 This pattern (and component) is used to bridge across an address space barrier between F´ deployments and route F´ port
6 calls to the remote deployment. It also receives incoming port calls from the remote component. Essentially, it is a
7 port multiplexer/demultiplexer that serializes the port calls to an `Fw::Buffer` object and outputs this data to an
8 implementation of the `Drv::ByteStreamDriverModel` for transport out of the current deployment's address space.
9 
10 **Note:** At this time, the hub drops send/receive errors.
11 Individual projects that need to monitor send/receive errors may need to add this functionality to the GenericHub [see idiosyncrasies](#idiosyncrasies)
12 
13 ## Design
14 
15 The generic hub accepts incoming port calls using an array of Serial input ports. This allows any port call to be
16 bridged across the connection. There is also an incoming set of ports supporting `Fw::Buffer` type allowing the generic
17 hub to send generic byte data across (e.g. file packets) the connection. These connections are serialized into the outgoing
18 connections to the driver. On the far side of the byte driver, another generic hub deserializes these port calls and buffer
19 calls into normal F´ port calls on the new deployment as if they were made locally.
20 
21 Generic hub inputs on one hub should be parallel to the outputs on the other hub and vise-versa.
22 It is also essential that users never pass a **pointer** into the generic hub, as pointer data will become invalid when it
23 leaves the current address space. A sample configuration is shown below. Finally, the generic hub is bidirectional, so it
24 can operate on inputs and produce outputs as long as its remote counterpart is hooked up in parallel.

52 ## Configuration
53 
54 Generic hub maximum output and input ports are configured using `AcConstants.ini` as shown below. Since hubs work in
55 tandem with another hub, the input ports on the first must match the output ports on the second. Both the number of port
56 and buffer inputs/outputs may be configured.
57 
58 ```ini
59 ; Hub connections. Connections on all deployments should mirror these settings.
60 GenericHubInputPorts = 10
61 GenericHubOutputPorts = 10
62 GenericHubInputBuffers = 10
63 GenericHubOutputBuffers = 10
64 ```
65 
66 The above configuration may be used with both deployments hubs as the input/output pairs match.

68 ## Idiosyncrasies
69 
70 Currently, the `Drv::ByteStreamDriverModel` can report errors and failures. This generic hub component drops these errors.
71 Users who expect the driver to error should adapt this component to handle this issue. Future versions of this component
72 may correct this issue by calling to a fault port on error.
