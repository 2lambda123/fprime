1 # F´ GPS Tutorial
2 
3 In this guide, we will cover the basics of working with F´ by attaching a GPS receiver to a serial driver and running
4 the application on a Raspberry PI. In order to fully benefit from this tutorial, the user should acquire any NMEA
5 compatible UART GPS receiver and a raspberry pi. In this tutorial, we use a NMEA GPS receiver with micro-USB such that
6 the code may be run on both the laptop and the Raspberry PI.
7 
8 In the DIY electronics community there is an abundance of cheap GPS receivers based around the NMEA protocol. These
9 receivers may support a USB interface pretending to be an ACM device for basic serial communication. The messages these
10 receivers send are NMEA formatted ASCII text.
11 
12 ![USB GPS Hardware](img/usb-gps.jpg)
13 
14 This tutorial will show how to integrate one of these GPS receivers with the F´ framework by wrapping it in a Component
15 and defining commands, telemetry, and log events. We will create a GpsApp deployment for the Component where it will
16 be wired to a standard UART driver in order to receive messages. Then we can cross compile it for the Raspberry PI and
17 run the application against the F´ ground system.
18 
19 **Note:** A completed version of this tutorial app is available [here](https://github.com/fprime-community/gps-tutorial)
20 for use as a demo or to help debug issues that come up when going through the tutorial.

50 ### Designing the GPS Component
51 
52 The F’ designs are specified by XML files that are processed by code generators to create C++ source and header files.
53 An XML file represents a single entity in the F´ system (Component, Port, Serializable, or Deployment). Command, Event,
54 and Telemetry Channel specifications are also written in XML. Further information is in the full F´ user guide.
55 [User Guide](../../UsersGuide/FprimeUserGuide.pdf). This application does not need any custom ports, as we are using the
56 standard ports to create our GPS handler. Custom ports can be seen in the [Math Component Tutorial](../MathComponent/Tutorial.md).
57 
58 In this section we will create a directory for our GPS component, and design the component through XML. The first step
59 to making the component is to make a project directory to hold our project, and a component subdirectory for our GPS.
60 
61 ```shell
62 cd fprime
63 mkdir -p GpsApp/Gps
64 cd GpsApp/Gps
65 ```
66 All files in this component section will be created in the `Gps` subdirectory.
67 
68 Next, in the GPS subdirectory, we will create a file called *GpsComponentAi.xml* filled with the below text. This
69 represents our component's design by defining the ports it uses to connect with other components and the files used to
70 specify commands, telemetry, and events. As can be seen, we are creating our component with 8 ports, the 4 functional
71 ports defined above, and 4 additional ports described below:
72 
73 1. **cmdIn**: an input port of *Fw::Cmd* type used to process commands sent to this component.
74 2. **cmdRegOut**: an output port of *Fw::CmdReg* type used to register this component's with the command dispatcher
75 3. **cmdResponseOut**: an output port of *Fw::CmdResponse* type used respond to dispatched commands
76 4. **eventOut**: an output port of *Fw::Log* type used to send events out
77 5. **textEventOut**: an output port of *Fw::LogText* type used to send events in a text form
78 6. **tlmOut**: an output port of *Fw::Tlm* type used to send out telemetry channels
79 7. **serialRecv**: an input port of type *Drv::SerialRead* used to receive serial data buffers,
80 8. **serialBufferOut**: an output port of type *Fw::BufferSend* used at startup to supply buffers to the serial driver
81 
82 Input ports are invoked from external components, and must be handled by the component. Output ports are used by this
83 component to invoke actions of other components. Here **cmdIn** and **serialRecv** are inputs for commands and serial
84 data respectively. Each will define a handler (seen later) to handle these invocations. All other ports are used to
85 invoke other components to send data buffers, events, telemetry, etc.
86 
87 The `GpsComponentAi.xml` file in the `Gps` subdirectory should look like:
88 ```xml
89 <?xml version="1.0" encoding="UTF-8"?>
90 <?xml-model href="../../Autocoders/schema/ISF/component_schema.rng" type="application/xml" schematypens="http://relaxng.org/ns/structure/1.0"?>
91 <!-- GPS Tutorial: GpsComponentAi.xml
92 
93 This is the design of GPS component. The goal is to read GPS messages from a UART port, and produce Events, and
94 Telemetry that represent the GPS link. This will also have a command to emit the lock status of the GPS signal. This
95 is an active component, meaning it will have it's own thead. It will therefore process messages at its own pace, and
96 will not need an external thread of execution to run on.
97 
98 It has 3 standard command ports, 2 standard event ports, 1 standard telemetry port, and 2 ports to interact with the
99 serial driver.
100 -->
101 
102 <component name="Gps" kind="active" namespace="GpsApp" modeler="true">
103  <!-- Import command ports -->
104  <import_port_type>Fw/Cmd/CmdPortAi.xml</import_port_type>
105  <import_port_type>Fw/Cmd/CmdRegPortAi.xml</import_port_type>
106  <import_port_type>Fw/Cmd/CmdResponsePortAi.xml</import_port_type>
107  <!-- Import event ports -->
108  <import_port_type>Fw/Log/LogPortAi.xml</import_port_type>
109  <import_port_type>Fw/Log/LogTextPortAi.xml</import_port_type>
110  <!-- Import telemetry ports -->
111  <import_port_type>Fw/Tlm/TlmPortAi.xml</import_port_type>
112  <!-- Import ports for serial driver -->
113  <import_port_type>Drv/SerialDriverPorts/SerialReadPortAi.xml</import_port_type>
114  <import_port_type>Fw/Buffer/BufferSendPortAi.xml</import_port_type>
115  <!-- Import command, telemetry, and event dictionaries -->
116  <import_dictionary>GpsApp/Gps/Commands.xml</import_dictionary>
117  <import_dictionary>GpsApp/Gps/Telemetry.xml</import_dictionary>
118  <import_dictionary>GpsApp/Gps/Events.xml</import_dictionary>
119 
120  <ports>
121  <!-- Command port definitions: command input receives commands, command reg out, and response out are
122  ports used to register with the command dispatcher, and return responses to it -->
123  <port name="cmdIn" data_type="Fw::Cmd" kind="input" role="Cmd" max_number="1">
124  </port>
125  <port name="cmdRegOut" data_type="Fw::CmdReg" kind="output" role="CmdRegistration" max_number="1">
126  </port>
127  <port name="cmdResponseOut" data_type="Fw::CmdResponse" kind="output" role="CmdResponse" max_number="1">
128  </port>
129  <!-- Event ports: send events, and text formated events -->
130  <port name="eventOut" data_type="Fw::Log" kind="output" role="LogEvent" max_number="1">
131  </port>
132  <port name="textEventOut" data_type="Fw::LogText" kind="output" role="LogTextEvent" max_number="1">
133  </port>
134  <!-- Telemetry ports -->
135  <port name="tlmOut" data_type="Fw::Tlm" kind="output" role="Telemetry" max_number="1">
136  </port>
137  <!-- Serial ports: one to receive serial data, and one to provide buffers for the serial driver to use -->
138  <port name="serialRecv" data_type="Drv::SerialRead" kind="async_input" max_number="1">
139  </port>
140  <port name="serialBufferOut" data_type="Fw::BufferSend" kind="output" max_number="1">
141  </port>
142  </ports>
143 </component>
144 ```
145 This file first imports all *Ai.xml files needed for each port type, imports our command, telemetry channel, and event
146 definition XMLs, and then defines all ports as we saw described above. There are several things to note:
147 
148 1. The GPS component is an *active* component, which has a thread of its own to execute on. This was chosen as the GPS
149 component has no realtime deadlines and is expected to run in parallel with other components in the system.
150 2. *async_input* is used for the input port from the serial driver. The handler should be run on the *active*
151 component's thread opposed to the invoking component's thread.
152 
153 *active* components with *async_input* ports are a fairly common initial design for components. They are typically used
154 unless the system has no thread scheduler, there are firm realtime deadlines, or other off-nominal requirements must be
155 met. Port and component types are described in more detail in the aforementioned User Guide.
156 
157 Instantiating the GPS component, and connecting it with other components in the system is done at the system level,
158 enabling the individual components to be reused in different applications. We will see this step later, after we design
159 our Commands, Events and Telemetry. We will also implement the C++ code as well.

161 ### Creating Commands.xml, Events.xml and Telemetry.xml Dictionaries
162 
163 These three XML dictionaries define the structure of commands, events, and telemetry that our component uses. This will
164 allow the autocoder to automatically generate the needed code to process commands, and emit events and telemetry. This
165 allows the developer to concentrate on the specific code for the component as opposed to hand coding the structure of
166 these entities.
167 
168 First we will create a command dictionary. The purpose of our command is to report the lock status of the GPS unit. This
169 command will trigger code to emit an event, which will report if the GPS has "locked" status or not. `Commands.xml` in
170 the `Gps` subdirectory should look like the following:
171 ```xml
172 <?xml version="1.0" encoding="UTF-8"?>
173 <?oxygen RNGSchema="file:../xml/ISF_Component_Schema.rnc" type="compact"?>
174 <!-- GPS Tutorial: GpsApp/Gps/Commands.xml
175 
176 This defines a single command to report the lock status of the GPS. This demonstrates a simple command that is useful
177 when working with GPS to determine if the data should be trusted.
178 -->
179 <commands>
180  <!-- Define a single command that runs asynchronously on the component's own thread. The opcode "0" is relative to
181  the GPS component's command space. The mnemonic is the string a user will use to refer to this command. -->
182  <command kind="async" opcode="0" mnemonic="Gps_ReportLockStatus" >
183  <comment>A command to force an EVR reporting lock status.</comment>
184  </command>
185 </commands>
186 ```
187 There are several notes to consider:
188 
189 1. We use an *async*s command for the same reason as we use *async_input* ports above.
190 2. Each component defines it's own set of opcodes indexed from 0. The autocoder will prevent collisions between
191 components by adding a prefix to the component's final opcode.
192 3. Users typically refer to the component's mnemonic and the opcode is typically internal to the F´ system.
193 
194 Next we will create an Events.xml dictionary that setup the events our component can emit. In this case we have two
195 events, GPS locked and GPS lock lost. The `Events.xml` file in the `Gps` subdirectory should look like:
196 
197 ```xml
198 <?xml version="1.0" encoding="UTF-8"?>
199 <?oxygen RNGSchema="file:../xml/ISF_Component_Schema.rnc" type="compact"?>
200 <!-- GPS Tutorial: GpsApp/Gps/Events.xml
201 
202 This defines two events, one at activity hi level to report that lock has been acquired, and one at warning hi level to
203 indicate lock lost.
204 -->
205 <events>
206  <event id="0" name="Gps_LockAquired" severity="ACTIVITY_HI" format_string="GPS lock acquired">
207  <comment>A notification on GPS lock acquired</comment>
208  </event>
209  <event id="1" name="Gps_LockLost" severity="WARNING_HI" format_string="GPS lock lost">
210  <comment>A warning on GPS lock lost</comment>
211  </event>
212 </events>
213 ```
214 Here it should be noted that:
215 1. *id*s are indexed per-component, like opcodes
216 2. Typically users refer to names of telemetry channels
217 3. Format strings are not downlinked, but stored for display purposes
218 
219 Finally, we should create a Telemetry.xml dictionary. It will specify that we will downlink GPS latitude, GPS longitude,
220 GPS altitude, GPS time, and current number of satellites visible to the GPS unit. These are all standard fields emitted
221 GPS units and are the heart of our application. Our `Telemetry.xml` file in the `Gps` subdirectory should look like:
222 
223 ```xml
224 <?xml version="1.0" encoding="UTF-8"?>
225 <?oxygen RNGSchema="file:../xml/ISF_Component_Schema.rnc" type="compact"?>
226 <!-- GPS Tutorial: GpsApp/Gps/Telemetry.xml
227 
228 This defines four telemetry channels to report basic GPS information.
229 -->
230 <telemetry>
231  <channel id="0" name="Gps_Latitude" data_type="F32" abbrev="GPS-0000">
232  <comment>The current latitude</comment>
233  </channel>
234  <channel id="1" name="Gps_Longitude" data_type="F32" abbrev="GPS-0001">
235  <comment>The current longitude</comment>
236  </channel>
237  <channel id="2" name="Gps_Altitude" data_type="F32" abbrev="GPS-0002">
238  <comment>The current altitude</comment>
239  </channel>
240  <channel id="3" name="Gps_Count" data_type="U32" abbrev="GPS-0003">
241  <comment>The current number of satilites</comment>
242  </channel>
243 </telemetry>
244 ```
245 Some notes:
246 1. *id*s are indexed per-component, like opcodes, and event ids
247 2. Typically users refer to names of telemetry channels, but a shorthand for display is the abbrev
248 3. Telemetry channels can be primitive types and serializable. All of the GPS data could be reimplemented as a single
249 serializable channel.

254 ## Setting Up the Build System for Gps and GpsApp
255 
256 Now it is time to create *CMakeList.txt* files for the GPS component, and GpsApp deployment. This will allow us to run
257 our GPS component through the autocoder, and receive implementation templates in order to save time/effort.
258 
259 First, in the `Gps` directory, create a module specific *CMakeLists.txt* file. Since we have not created any C++ files,
260 our *CMakeLists.txt* will only contain the Ai.xml file we created. We'll add C++ files once we have created them.
261 
262 The `CMakeLists.txt` created in the `Gps` directory should look like this:
263 ```

268 # MOD_DEPS: (optional) module dependencies

275 )

365 ## Coding Our Module
366 
367 Now it is time to code our module to read the GPS module and downlink the GPS telemetry. This is where the framework
368 will help us considerably. All these previous steps set us up to use the autocoding features of F´. We can generate the
369 basic implementation of the code by using the `fprime-util` *impl* command, which generates needed
370 *GpsComponentImpl.cpp-tmpl* and *GpsComponentImpl.hpp-tmpl* files. We can use these as the basis for our implementation.
371 In addition, the framework will also generate * *Ac.?pp* files, which handle the work of connecting ports allowing us
372 to write minimal code to support the component interface. First we generate code templates, and move them into place.
373 Since we don't already have implementations we can safely rename the template files without first checking for existing
374 files.
375 
376 Change back to the `Gps` subdirectory and generate implementations with:
377 ```shell
378 cd Gps
379 fprime-util impl
380 mv GpsComponentImpl.cpp-template GpsComponentImpl.cpp
381 mv GpsComponentImpl.hpp-template GpsComponentImpl.hpp
382 ```
383 *Note:* If the developer regenerates the templates, care must be taken to not overwrite already implemented code by
384 copying the templates to the implementation files.

418 ### Implementation
419 
420 In the generated implementations, we can seen that we have two actions "TODO". First we will need to implement a
421 function called *serialRecv_handler* and the second is to implement a command handler for
422 *Gps_ReportLockStatus_cmdHandler*. The other functions of our code are provided as functions we can use when we
423 implement these two pieces. Those available functions are described below:
424 
425 1. log_ACTIVITY_HI_Gps_LockAquired: used to emit the event *Gps_Lock_aquired* as defined in Events.xml
426 2. log_WARNING_HI_Gps_LockLost: used to emit the event *Gps_LockLost* as defined in Events.xml
427 3. tlmWrite_Gps_Latitude: used to send down *Latitude* telemetry as defined in Telemetry.xml
428 4. tlmWrite_Gps_Longitude: used to send down *Longitude* telemetry as defined in Telemetry.xml
429 5. tlmWrite_Gps_Altitude: used to send down *Altitude* telemetry as defined in Telemetry.xml
430 6. tlmWrite_Gps_Count: used to send down *Count* telemetry as defined in Telemetry.xml
431 7. sendCommandResponse: used to respond to a sent command. Call this in the above cmdHandler.
432 
433 In order to make a GPS processor that works well, we need to take the following steps:
434 
435 0. Create a preamble function and send some buffers for the serial driver to use
436 1. Implement the serialRecv_handler function (called by the serial driver with one of the above buffers)
437 2. Break down the GPS message
438 3. Return the buffer to the serial driver
439 4. Downlink telemetry in serialRecv_handler
440 5. If lock is newly found, downlink a *LockAcquired* EVR
441 6. If lock is newly lost, downlink a *LockLost* EVR
442 7. Downlink a *LockAcquired* EVR in commandHandler, if lock is currently held
443 8. Downlink a *LockLost* EVR in commandHandler, if lock is not currently held
444 9. Respond to the commandHandler with a sendCommandResponse call
445 
446 These steps are called out in the following implementations of these two files. Since the purpose of this tutorial
447 is not to demonstrate how to write each line of code, the steps above are called out in comments in the code.
448

497  //Step 0: The linux serial driver keeps its storage externally. This means that we need to supply it some buffers to
498  // work with. This code will loop through our member variables holding the buffers and send them to the linux
499  // serial driver. 'preamble' is automatically called after the system is constructed, before the system runs
500  // at steady-state. This allows for initialization code that invokes working ports.
501  void GpsComponentImpl :: preamble(void)
502  {
503  for (NATIVE_INT_TYPE buffer = 0; buffer < NUM_UART_BUFFERS; buffer++) {
504  //Assign the raw data to the buffer. Make sure to include the side of the region assigned.
505  this->m_recvBuffers[buffer].setdata((U64)this->m_uartBuffers[buffer]);
506  this->m_recvBuffers[buffer].setsize(UART_READ_BUFF_SIZE);
507  // Invoke the port to send the buffer out.
508  this->serialBufferOut_out(0, this->m_recvBuffers[buffer]);
509  }
510  }
511 
512  GpsComponentImpl ::
513  ~GpsComponentImpl(void)
514  {
515 
516  }
517 
518  // ----------------------------------------------------------------------
519  // Handler implementations for user-defined typed input ports
520  // ----------------------------------------------------------------------
521 
522  // Step 1: serialIn
523  //
524  // By implementing this "handler" we can respond to the serial device sending us data buffers containing the GPS
525  // data. This handles our serial messages. It should perform the actions we expect from the design phases.
526  void GpsComponentImpl ::
527  serialRecv_handler(
528  const NATIVE_INT_TYPE portNum, /*!< The port number*/
529  Fw::Buffer &serBuffer, /*!< Buffer containing data*/
530  Drv::SerialReadStatus &serial_status /*!< Status of read*/
531  )
532  {
533  // Local variable definitions
534  int status = 0;
535  float lat = 0.0f, lon = 0.0f;
536  GpsPacket packet;
537  // Grab the size (used amount of the buffer) and a pointer to the data in the buffer
538  U32 buffsize = static_cast<U32>(serBuffer.getsize());
539  char* pointer = reinterpret_cast<char*>(serBuffer.getdata());
540  // Check for invalid read status, log an error, return buffer and abort if there is a problem
541  if (serial_status != Drv::SER_OK) {
542  Fw::Logger::logMsg("[WARNING] Received buffer with bad packet: %d\n", serial_status);
543  // We MUST return the buffer or the serial driver won't be able to reuse it. The same buffer send call is used
544  // as we did in "preamble". Since the buffer's size was overwritten to hold the actual data size, we need to
545  // reset it to the full data block size before returning it.
546  serBuffer.setsize(UART_READ_BUFF_SIZE);
547  this->serialBufferOut_out(0, serBuffer);
548  return;
549  }
550  // If not enough data is available for a full messsage, return the buffer and abort.
551  else if (buffsize < 24) {
552  // We MUST return the buffer or the serial driver won't be able to reuse it. The same buffer send call is used
553  // as we did in "preamble". Since the buffer's size was overwritten to hold the actual data size, we need to
554  // reset it to the full data block size before returning it.
555  serBuffer.setsize(UART_READ_BUFF_SIZE);
556  this->serialBufferOut_out(0, serBuffer);
557  return;
558  }
559  //Step 2:
560  // Parse the GPS message from the UART (looking for $GPGGA messages). This uses standard C functions to read all
561  // the defined protocol messages into our GPS package struct. If all 9 items are parsed, we break. Otherwise we
562  // continue to scan the block of data looking for messages further in.
563  for (U32 i = 0; i < (buffsize - 24); i++) {
564  status = sscanf(pointer, "$GPGGA,%f,%f,%c,%f,%c,%u,%u,%f,%f",
565  &packet.utcTime, &packet.dmNS, &packet.northSouth,
566  &packet.dmEW, &packet.eastWest, &packet.lock,
567  &packet.count, &packet.filler, &packet.altitude);
568  //Break when all GPS items are found
569  if (status == 9) {
570  break;
571  }
572  pointer = pointer + 1;
573  }
574  //If we failed to find the GPGGA then return the buffer and abort.
575  if (status == 0) {
576  // We MUST return the buffer or the serial driver won't be able to reuse it. The same buffer send call is used
577  // as we did in "preamble". Since the buffer's size was overwritten to hold the actual data size, we need to
578  // reset it to the full data block size before returning it.
579  serBuffer.setsize(UART_READ_BUFF_SIZE);
580  this->serialBufferOut_out(0, serBuffer);
581  return;
582  }
583  // If we found some of the message but not all of the message, then log an error, return the buffer and exit.
584  else if (status != 9) {
585  Fw::Logger::logMsg("[ERROR] GPS parsing failed: %d\n", status);
586  // We MUST return the buffer or the serial driver won't be able to reuse it. The same buffer send call is used
587  // as we did in "preamble". Since the buffer's size was overwritten to hold the actual data size, we need to
588  // reset it to the full data block size before returning it.
589  serBuffer.setsize(UART_READ_BUFF_SIZE);
590  this->serialBufferOut_out(0, serBuffer);
591  return;
592  }
593  //GPS packet locations are of the form: ddmm.mmmm
594  //We will convert to lat/lon in degrees only before downlinking
595  //Latitude degrees, add on minutes (converted to degrees), multiply by direction
596  lat = (U32)(packet.dmNS/100.0f);
597  lat = lat + (packet.dmNS - (lat * 100.0f))/60.0f;
598  lat = lat * ((packet.northSouth == 'N') ? 1 : -1);
599  //Longitude degrees, add on minutes (converted to degrees), multiply by direction
600  lon = (U32)(packet.dmEW/100.0f);
601  lon = lon + (packet.dmEW - (lon * 100.0f))/60.f;
602  lon = lon * ((packet.eastWest == 'E') ? 1 : -1);
603  //Step 4: call the downlink functions to send down data
604  tlmWrite_Gps_Latitude(lat);
605  tlmWrite_Gps_Longitude(lon);
606  tlmWrite_Gps_Altitude(packet.altitude);
607  tlmWrite_Gps_Count(packet.count);
608  //Lock status update only if changed
609  //Step 5,6: note changed lock status
610  // Emit an event if the lock has been acquired, or lost
611  if (packet.lock == 0 && m_locked) {
612  m_locked = false;
613  log_WARNING_HI_Gps_LockLost();
614  } else if (packet.lock == 1 && !m_locked) {
615  m_locked = true;
616  log_ACTIVITY_HI_Gps_LockAquired();
617  }
618  // We MUST return the buffer or the serial driver won't be able to reuse it. The same buffer send call is used
619  // as we did in "preamble". Since the buffer's size was overwritten to hold the actual data size, we need to
620  // reset it to the full data block size before returning it.
621  serBuffer.setsize(UART_READ_BUFF_SIZE);
622  this->serialBufferOut_out(0, serBuffer);
623  }
624 
625  // ----------------------------------------------------------------------
626  // Command handler implementations
627  // ----------------------------------------------------------------------
628  //Step 7,8: respond to a command to report lock status.
629  //
630  // When a status command is received, respond by emitting the
631  // current lock status as an Event.
632  void GpsComponentImpl ::
633  Gps_ReportLockStatus_cmdHandler(
634  const FwOpcodeType opCode,
635  const U32 cmdSeq
636  )
637  {
638  //Locked-force print
639  if (m_locked) {
640  log_ACTIVITY_HI_Gps_LockAquired();
641  } else {
642  log_WARNING_HI_Gps_LockLost();
643  }
644  //Step 9: complete command
645  this->cmdResponse_out(opCode, cmdSeq, Fw::COMMAND_OK);
646  }
647 } // end namespace GpsApp

836 ### Build the Topology Sources
837 
838 A topology consists of several files that instantiate the components in C++, construct and initialize the system, and
839 represent the "main function" or entry point into the code. These files are shown below:
840 
841 1. **GpsTopologyAppAi.xml**: the design file showing the list of components, and the connections between components.

847 4. **Main.cpp**: entry point to the code.

850 initialization code. All of these files are referenced by the CMake files we inherited from the reference app. Building
851 the distribution (fprime/GpsApp) will include the topology (GpsApp/Top) as its entry-point creating a single binary,
852 which represents our software.

941 ## Cross Compiling for the Raspberry PI
942 
943 Once we have the code building for Linux, it is time to engage the raspberry pi cross compilation tools in order to
944 build a binary that will run on the Raspberry PI. We'll be working with code for the Raspberry PI 3, but it should work
945 on the RPI 2 as well.
946 
947 ### Cross Compiling Explicitly
948 
949 The first step is to follow the installation of the tools for the RPI cross compile as documented here:
950 [RPI Deployment README.md](../../../RPI/README.md).
951 
952 In order to cross-compile for the a specific architecture, the user needs to generate a new build directory using a
953 toolchain file for that architecture. F´ includes a toolchain for the raspberry PI, assuming the tools are installed in
954 the manner described in the above readme. This toolchain is called "raspberrypi". The build can be generated by running
955 the following commands in the `GpsApp` directory:
956 
957 ```shell
958 fprime-util generate raspberrypi
959 fprime-util install raspberrypi
960 ```
961 
962 This will generate the binary at `GpsApp/bin/arm-linux-gnueabihf/GpsApp`. The user then may run the ground system as
963 before. Ensure that the system and network firewall allow through port 50000 from the PI to the host, and then run:
964 
965 ```shell
966 fprime-gds -d . -n
967 ```
968 
969 Assuming there is no firewall or other network limits between the PI and the host, the user can run the following from a
970 separate terminal run the following:
971 
972 ```shell
973 cd fprime/GpsApp
974 scp bin/arm-linux-gnueabihf/GpsApp pi@<raspberry pi IP>
975 ssh pi@<raspberry pi IP>
976 ./GpsApp -a <ground system IP> -p 50000 -d <serial port, /dev/ttyACM0 for USB>
977 ```
978 
979 Now the PI should be powered up and running the embedded Linux binary, and our host system should be running the ground
980 system. If the code work when running natively, and isn't connecting for this example the cause it likely a firewall
981 issue. Again make sure port 50000 is exposed to the PI, and that the pi can ping the ground system machine.
982 
983 ### Setting the Cross Compile Build as Default
984 
985 As we saw above, cross compile builds can be done explicitly by setting the toolchain. However, some users may wish to
986 make this the default, and not need to specify it. This can be done by adding the following line to the deployment

988 call.

990 ```

992 ```

997 ```

999 fprime-util purge raspberrypi
1000 ```
