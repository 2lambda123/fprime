1 # Sequencing In F´
2 
3 F´ supports a very basic sequence format that allows for executing F´ commands at absolute times and at times relative
4 to previous commands. The sequence uses the F´ dictionary to translate the human readable sequence file into a minimized
5 binary format for upload and running using the command sequencer.
6 
7 Compiling this sequence is done with the `fprime-seqgen` utility and can be run once uploaded by issuing the `*.CS_RUN`
8 command of any command sequencer instance defined in the flight system.
9 
10 ## Writing an F´ Sequence File
11 
12 F´ sequence files consist of lists of commands. These commands start with a time argument, followed by the command
13 mnemonic, and lastly any arguments to command. See the example below, which are all pulled from the example sequence
14 file found here: [simple_sequence.seq](https://github.com/nasa/fprime/blob/devel/Gds/examples/simple_sequence.seq).
15 
16 ```
17 A2015-075T22:32:40.123 cmdDisp.CMD_NO_OP
18 ```
19 **Note:** see [Time Details](#time-details) for more on time formats and time bases.
20 
21 Here an absolute command was chosen. Times can be specified in in absolute or relative time format. An absolute time
22 starts with an `A` and specifies a calendar time. A relative command is specified starting with an `R`, which runs
23 relative to the previous command or start of the sequence. A relative command with an argument is shown below.
24 
25 ```
26 R01:00:01.050 CMD_NO_OP_STRING "Awesome string!" ; And a nice comment too
27 ```
28 
29 A list of these commands can be specified in a text file typically ending with the `.seq` extension. Comments start
30 with a ;. The example file above goes into greater explanation of the sample commands.
31 
32 ## Compiling A Sample Sequence
33 
34 The `fprime-seqgen` command can compile the sequence into a binary format that F´ flight software can execute. To do
35 this the user should provide a path to the flight software dictionary and a path to the text file discussed above.
36 Below is an example on how to run the sample example sequence with the Ref dictionary. Remember to build first or the
37 dictionary will not be generated.
38 
39 ```
40 fprime-seqgen fprime/Gds/examples/simple_sequence.seq -d fprime/Ref/build-artifacts/*/dict/RefTopologyAppDictionary.xml
41 ```
42 
43 Here the output file is not specified, so it will be a new file in the same directory as the sequence but ending with
44 the `.bin` extension.
45 
46 This binary file should be uploaded to the flight software. Given limitations on file path length, it should be copied
47 to `/tmp` when running locally.
48 
49 ## Running The Binary Sequence
50 
51 Once uploaded, the sequence can be run by executing the .CS_RUN command of any command sequencer. Here we will run the
52 `cmdSeq` instance's .CS_RUN command using an upload location of /tmp/sample_sequence.bin. This should work in the `Ref`
53 application.
54 
55 ```
56 cmdSeq.CS_RUN "/tmp/sample_sequence.bin"
57 ```
58 
59 **Note:** the sample sequence will run for multiple hours due to the specification of the relative commands.
60 
61 ## Time Details
62 
63 Time is represented in the sequence file using several formats. Relative times are represented with ISO_8601 time with
64 only subseconds being optional, see table below. Absolute times are represented in ISO_8601 using ordinal dates
65 (day of year) and time with optional subseconds.
66 
67 | | Format | Description | Example |
68 |---|---|---|---|
69 | Relative Times | RHH:MM:SS[.sss] | 'R' followed by ISO_8601 hour minute section and optional subseconds | R23:02:01.010 |
70 | Absolute Times | AYYYY-DDDTHH:MM:SS[.sss] | 'A' followed by ISO_8601 ordinal datetime format with optional subseconds | A2020-192T23:02:01.010 |
71 
72 **Note:** relative times cannot exceed 24 hours.
73 
74 Since sequences can specify absolute time it may be dangerous to run a sequence should the flight software time not be
75 synchronized with a known source. For example, it may be dangerous to run a sequence near system boot before time has
76 been polled from a hardware clock or system time source.
77 
78 A sequence can be built with an expected time base and said sequence will not run should the flight software report time
79 system in a different time base. For example, a sequence using TB_SC_TIME (spacecraft time) could be prevented from
80 running if the flight software is currently using raw processor time and has not synchronized time with a known time
81 source. Available time sources are below:
82 
83 | Timebase | Value | Explanation |
84 |---|---|---|
85 | TB_PROC_TIME | 1 | Sequence will run when only raw processor time is reported |
86 | TB_WORKSTATION_TIME | 2 | Sequence will run when time is synchronized with test workstation (for testing) |
87 | TB_SC_TIME | 3 | Sequence will run when time is synchronized with spacecraft time |
88 | TB_FPGA_TIME | 4 | Sequence will run when time is synchronized with FPGA/hardware clock |
89 | TB_DONT_CARE | 0xFFFF | Sequence will run regardless of flight software timebase |
90 
91 **Note:** the above descriptions represent typically usages of these time bases but are project specific i.e.
92 TB_SC_TIME might derive time from an internet time source.
