2 # Byte Stream Driver Model
3 
4 The byte stream driver is a generic model for drivers implementing a "stream of bytes" interface. Typically these
5 drivers operate with an outgoing stream and an incoming stream. The outgoing stream is represented by the "send" port
6 and the incoming stream is either polled using the "poll" port or return asynchronously via the "readCallback" port.
7 
8 ## Design
9 
10 The manager component (typically the ground interface) initiates the transfer of send data by calling the "send" port.
11 The caller will provide a `Fw::Buffer` containing the data to send and the port call will return a status of that send.
12 These responses are an enumeration whose values are described in the following table:
13 
14 | Value | Description |
15 |---|---|
16 | Drv::SEND_OK | Send functioned normally. |
17 | Drv::SEND_RETRY | Send should be retried, but a subsequent send should return SEND_OK. |
18 | Drv::SEND_ERROR | Send produced an error, future sends likely to fail. |
19 
20 **Note:** in either formation described below, send will operate as described here.
21 
22 ### Callback Formation
23 
24 ![Callback](./img/canvas-callback.png)
25 
26 In the callback formation, the byte stream driver component initiates the transfer of received data by calling the
27 "readCallback" output port. This port transfers any read data in a `Fw::Buffer` along with a status for the receive.
28 This status is an enumeration whose values are described in the following table:
29 
30 | Value | Description |
31 |---|---|
32 | Drv::RECV_OK | Receive functioned normally buffer contains valid data. |
33 | Drv::RECV_ERROR | Receive produced an error and buffer contains no valid data. |
34 
35 The following components implement the byte stream model using a callback formation:
36 - `DrvTcpClient`: a F´ component wrapper of the tcp client
37 - `DrvTcpServer`: a F´ component wrapper of the tcp server
38 - `DrvUdp`: a F´ component wrapper of the udp
39 
40 ### Polling Formation
41 
42 ![Poll](./img/canvas-poll.png)
43 
44 In the callback formation, The manager component (typically the ground interface) initiates the transfer of received
45 data by calling the "poll" input port. This port fills in the provided `Fw::Buffer` along with a status for the poll.
46 This status is an enumeration whose values are described in the following table:
47 
48 | Value | Description |
49 |---|---|
50 | Drv::POLL_OK | Poll functioned normally buffer contains valid data. |
51 | Drv::POLL_RETRY | Poll should be retried and a subsequent send should return POLL_OK. |
52 | Drv::POLL_ERROR | Poll produced an error and buffer contains no valid data. |
53 
54 **Note:** there are no known implementers of the polling formation, although this formation is best suited for
55 implementations running on baremetal machines.
