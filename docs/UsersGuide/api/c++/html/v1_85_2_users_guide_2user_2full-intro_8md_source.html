12 ## The Origins of F´
13 
14 F´ is an embedded systems framework developed build to meet certain goals expanded upon below.
15 These goals it particularly well suited for small embedded missions (and projects) that need
16 reliable components and infrastructure provided from the framework in
17 order to minimize development cost, schedule, or effort. These missions
18 include CubeSats, small-sats, and deployables. Despite being build for NASA missions, F´ can be used for any
19 embedded system regardless of the project size or field.
20 
21 
22 F´ was built to meet the following goals:
23 
24  - Capture the reusability of embedded projects into a Framework
25  - Ease the separating and reassembling of shareable system’s components
26  - Isolate components for ease of testing
27  - Adapt easily to new contexts
28  - Port to new architectures and platforms
29  - Be easy to use
30  - Scale and configure to meed new usse cases
31  - Perform well in resource constrained context
32 
33 F´ was developed at NASA's Jet Propulsion Laboratory for small-scale embedded flight software. These small missions
34 desperately needed a Flight-Quality framework to expedite development while reducing cost.

36 ## A (Brief) Introduction to Space Systems
37 
38 Although, F´ can be used to model any embedded system, it has its origin in the stars. It was built at NASA's Jet
39 Propulsion Laboratory for use in small-scale space flight missions and thus to understand the basics of these systems
40 is to more completely understand F´ itself.
41 
42 What is a space system? For the purposes of this guide, it is any embedded system in space. Typically these systems
43 compose the complete control and operation of a spacecraft to accomplish some mission. At JPL these systems are
44 typically conducting some aspect of scientific research.
45 
46 Since F´ is usually the complete software for one of these systems, F´ decomposes its projects into discreet
47 **Components** that each manage one part of the system. e.g. a Radio Component may control the radio hardware to
48 facilitate communication. **Components** are connected to one another via **Ports**. **Ports** allow communication
49 between **Components**. The complete graph or network of **Components** connected via **Ports** is called a
50 **Topology**, which encapsulates the full-system.
51 
52 F´ was built to support **Command and Data Handling** (C&DH) of these space systems as well as instruments running as
53 part of these space systems. This means F´ out-of-the-box is designed to handle commands sent from the ground, and
54 respond with telemetry to the ground. In F´ this telemetry is broken into **Events** representing history of actions
55 taken by the system, and channels representing the current state of the system broken into named channels that each
56 contain a portion of the state. e.g. an **Evnt** might be "Established Communications" and a **Channel** might be
57 "Current Temperature: 3C".
58 
59 All projects using F´ are composed of **Components**, **Ports**, and **Topologies**. Although F´ does not require the
60 the user to control the system using **Commands**, **Events**, and **Channels**, these constructs represent the typical
61 use case for F´projects and such constructs are built-in. Thus helpful to understand both **Components**, **Ports**, and
62 **Topologies**, as well as **Command**, **Events**, and **Channels** in order to fully understand the power of the F´
63 framework.

65 ## The Organization of F´ Deployments
66 
67 The core F′ software framework allows projects to be decomposed into a set of **Components** that are interconnected by
68 **Ports**. Each **Component** represents one discreet piece of the system. e.g. the Command Dispatcher is a framework
69 component used to dispatch incoming commands to be handled by another component in the system. Its job is to receive a
70 ground communication and translate that into an action, dispatch the action to another component, and await the
71 completion of this action. It emits **Events** to signify when the action is dispatched, and when it has completed. It
72 has **Channels** counting the number of commands dispatched.
73 
74 From this example we can see into the organization of an F´ project. **Components** are the key to system
75 modularization and each component has a set of **Ports** to communicate from. In addition, each **Component** may
76 define a set of **Commands** it may handle (these are the actions seen dispatched above), and each **Component** may
77 define a set of **Events** it can report as well as a set of **Channels** that it will send out. When a system is built
78 from these modules, the functionality is distributed amongst the **Components** and the **Topology** sets up the
79 communication such that the system can function.

81 ## Threads, Multi-Core Architectures and F´
82 
83 F´ was built for use on platforms running an Operating System (OS) and executing on a single core. Notably these systems
84 come with a thread scheduler. That being said it is entirely possible to use F´ on a baremetal system, or a multi-core
85 system, however; some care should be take when designing for such systems an understanding of execution context is
86 required. See: [F´ On Baremetal and Muti-Core Systems](../dev/baremetal-multicore.md)

88 ## Conclusion
89 
90 The F′ software framework is released as open source and has been ported to Linux, MacOS, Windows (WSL), VxWorks, ARINC
91  653, RTEMS, Baremetal(No OS), PPC, Leon3, x86, ARM (A15/A7), and MSP430. Mature sets of CD&H components are available
92 following flight process, such as code inspections, static analysis, and full-coverage unit testing.
